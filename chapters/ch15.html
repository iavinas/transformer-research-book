Chapter 15

Evaluating Language Models

A model that scores 93% on MMLU in 2026 tells you almost nothing — a dozen frontier models crowd the same narrow band, and the benchmark has appeared in countless pre-training corpora. The real question is not "how good is this model?" but "how do we know?" This chapter maps the full evaluation landscape: from perplexity, through the wave of saturated-and-replaced benchmarks, to the newest frontiers of agentic evaluation, reasoning-model assessment, and contamination-resistant dynamic benchmarks. By the end you will understand not just what each benchmark measures, but why the community keeps building new ones and how to choose the right evaluation for your specific question about a model.

Intrinsic Metrics

Before benchmarks, before leaderboards, the simplest question you can ask a language model is: how surprised are you by this text? Intrinsic metrics answer exactly that question — they measure how well the model's probability distribution matches the actual distribution of tokens in a held-out corpus. No downstream task, no human judges, just raw predictive quality.

Perplexity

Perplexity is the exponential of the average negative log-likelihood per token. Given a sequence of $T$ tokens $x_1, x_2, \ldots, x_T$, the model assigns a probability $P(x_t \mid x_{<t})$ to each token conditioned on its predecessors. Perplexity is:

$$\text{PPL} = \exp\!\left(-\frac{1}{T}\sum_{t=1}^{T} \log P(x_t \mid x_{<t})\right)$$

The $\log P(x_t \mid x_{<t})$ term is the log-probability the model assigns to the correct next token (a scalar). The sum accumulates these across all $T$ positions, and dividing by $T$ gives the average per-token log-likelihood. Exponentiating converts this back to an interpretable scale: a perplexity of 20 means the model is, on average, as uncertain as if it were choosing uniformly among 20 equally likely tokens at each step.

Lower perplexity is better. A perfect model that assigns probability 1 to every correct token achieves perplexity 1. A model choosing uniformly from a vocabulary of 50,000 tokens would have perplexity 50,000. In practice, GPT-2 achieved perplexity ~22 on WikiText-103; GPT-3 brought this below 10; modern frontier models push it below 5 on clean English text.

The following code computes perplexity from model logits over a token sequence.


import torch
import torch.nn.functional as F

def compute_perplexity(logits, target_ids):
    """
    Compute perplexity from model output logits.

    Args:
        logits: (batch, seq_len, vocab_size) — raw model output
        target_ids: (batch, seq_len) — ground-truth token ids
    Returns:
        Scalar perplexity value.
    """
    # Shift: predict token t+1 from position t
    shift_logits = logits[:, :-1, :]        # (batch, seq_len-1, vocab_size)
    shift_targets = target_ids[:, 1:]       # (batch, seq_len-1)

    # Cross-entropy per token (no reduction — we need the per-token values)
    log_probs = F.log_softmax(shift_logits, dim=-1)  # (batch, seq_len-1, vocab_size)
    token_log_probs = log_probs.gather(
        dim=-1,
        index=shift_targets.unsqueeze(-1)            # (batch, seq_len-1, 1)
    ).squeeze(-1)                                     # (batch, seq_len-1)

    # Average negative log-likelihood across all tokens
    avg_nll = -token_log_probs.mean()  # scalar
    perplexity = torch.exp(avg_nll)    # scalar
    return perplexity.item()

# Example: random logits over a 10-token sequence, vocab size 1000
torch.manual_seed(42)
logits = torch.randn(1, 10, 1000)     # (1, 10, 1000)
targets = torch.randint(0, 1000, (1, 10))  # (1, 10)
ppl = compute_perplexity(logits, targets)
print(f"Perplexity: {ppl:.1f}")        # ~1000 for random logits (uniform over vocab)

Bits-per-character and bits-per-byte

Perplexity depends on the tokenizer. A model with a 32,000-token vocabulary and a model with a 128,000-token vocabulary are playing different games — the second model's tokens carry more information each, so its perplexity will be lower even if its underlying language understanding is identical. To compare across tokenizers, we convert to bits-per-character (BPC) or bits-per-byte (BPB):

$$\text{BPC} = \frac{1}{C} \sum_{t=1}^{T} \left(-\log_2 P(x_t \mid x_{<t})\right)$$

where $C$ is the total character count in the sequence (a scalar count, not a tensor). BPB replaces $C$ with the byte count $B$. Both are tokenizer-agnostic because they normalize by the actual content length rather than the number of tokens. A model achieving 0.9 BPB on English text is compressing each byte to less than one bit of information — a strong signal of linguistic understanding.

Why low perplexity does not guarantee good generation

Perplexity measures how well a model predicts text from a held-out corpus. It does not measure whether the model can produce useful, coherent, or safe text when generating freely. A model can achieve excellent perplexity by memorizing common patterns while still generating repetitive loops, hallucinating facts, or ignoring instructions. The gap between "predicts well" and "generates well" is real and widening: instruction-tuned models often have slightly worse perplexity on raw web text than their base counterparts, yet are dramatically better at following human intent. Perplexity is necessary but not sufficient — the rest of this chapter covers what else you need.

Knowledge and Reasoning Benchmarks

The history of LLM benchmarks is a history of rapid saturation. Each generation of benchmarks is designed to be hard, dominates evaluation for a few years, then collapses into irrelevance as models surpass expert-level performance and the benchmark loses its ability to discriminate. Understanding this cycle is essential for interpreting any leaderboard.

The saturation cycle

GLUE (2018) tested basic natural language understanding: sentiment analysis, textual entailment, paraphrase detection. It was saturated within a year. SuperGLUE (2019) raised the bar with harder tasks and was itself surpassed by 2021. MMLU (2021) — 57 academic subjects, 15,908 multiple-choice questions evaluated in zero-shot or 5-shot settings — held up longer, but by 2025 frontier models exceeded 90%, compressing the entire useful range into a few percentage points where noise dominates signal.

The pattern is clear: each benchmark buys the community 2-4 years of useful signal before saturation makes it a checkbox rather than a measurement. This drives the design of every benchmark discussed below.

MMLU-Pro

TIGER-AI-Lab released MMLU-Pro at NeurIPS 2024 as a direct response to MMLU's collapse. The key changes: 12,000 graduate-level questions with 10 answer choices instead of 4, dramatically reducing the chance of guessing correctly (from 25% to 10%). Accuracy drops 16-33% compared to MMLU across the same models. A revealing finding: chain-of-thought prompting now helps significantly on MMLU-Pro, unlike on original MMLU where it made little difference — the questions genuinely require multi-step reasoning. Prompt sensitivity also drops from 4-5% on MMLU to about 2% on MMLU-Pro, making scores more reproducible. Even so, frontier models are already approaching 90% on MMLU-Pro by early 2026.

GPQA-Diamond

GPQA (Graduate-Level Google-Proof Q&A) takes a different approach to hardness. Rather than more choices, it uses expert-written questions in biology, physics, and chemistry that are deliberately "Google-proof" — meaning that a skilled non-expert with unrestricted internet access scores only 34%, while domain PhD experts score 65-74%. The Diamond subset (198 questions) has become a standard benchmark for frontier models. The design principle is powerful: if web search cannot help a smart generalist answer the question, then a model must possess genuine domain reasoning to get it right.

Humanity's Last Exam

In January 2025, the Center for AI Safety and Scale AI released what they called the final closed-ended academic benchmark. Humanity's Last Exam (HLE) comprises 2,500 questions across 100+ subjects, contributed by roughly 1,000 experts from 500+ institutions in 50 countries. When released, the best model (o1) scored just 8%. Published in Nature in 2025, HLE was designed to remain challenging for years rather than months. As of early 2026, the top score sits around 45% — still far from human expert performance of ~90% within their domains. The 14% of questions requiring diagram comprehension expose another weakness: many models struggle when visual reasoning intersects domain knowledge.

SimpleQA

OpenAI's SimpleQA (November 2024) attacks the problem from a different angle: instead of asking hard questions, it asks easy ones — 4,326 short, fact-seeking questions with single indisputable answers — and measures whether the model hallucinates or says "I don't know." GPT-4o scored below 40%. SimpleQA directly targets the trust problem: if a model confidently answers "What year did X happen?" incorrectly, users cannot trust it on harder questions. The benchmark also measures calibration — whether the model knows what it knows.

Commonsense reasoning suite

HellaSwag (sentence completion), ARC (grade-school science), and WinoGrande (coreference resolution) remain widely reported but are largely saturated at the frontier (all above 95%). They still serve a purpose for evaluating smaller or open-source models where performance gaps remain meaningful. BIG-Bench Hard — the 23 most challenging tasks from the original 200+ task BIG-Bench suite — retains some discriminative power, particularly on tasks involving logical deduction and multi-step reasoning.

Mathematics and Formal Reasoning Benchmarks

Mathematical reasoning occupies a privileged position in LLM evaluation: answers are unambiguously correct or incorrect, problems scale in difficulty from arithmetic to open research questions, and chain-of-thought reasoning is both visible and verifiable. The saturation pattern has played out here too, but faster — and the replacements are more dramatic.

MATH and MATH-500

The MATH benchmark (Hendrycks et al., 2021) contains 12,500 competition-level problems across algebra, number theory, geometry, and more. When released, the best model scored 7%. By 2025, frontier reasoning models exceed 90%. MATH-500, a curated 500-problem subset, is now commonly used for efficiency but suffers from the same ceiling compression.

AIME 2025

The American Invitational Mathematics Examination provides fresh problems annually — 30 problems from the 2025 AIME I and II editions, each requiring an integer answer between 000 and 999. AIME problems demand multi-step algebraic and combinatorial reasoning beyond simple pattern matching. By early 2026, the best model achieves a perfect 100% on AIME 2025, and over 105 models have been evaluated on this benchmark. The average score across all models is 78.3%, showing it still differentiates across the broader model landscape even as the frontier has saturated it.

FrontierMath

Epoch AI released FrontierMath in November 2024: 350 original problems across four difficulty tiers, developed with 60+ mathematicians including Fields Medalists Terence Tao, Timothy Gowers, and Richard Borcherds. Tier 1 problems are challenging undergraduate exercises; Tier 4 problems connect to active research and typically require hours or days for expert mathematicians to solve. When released, no AI model solved more than 2% of problems. By March 2026, the best model scores around 50% on Tiers 1-3 and 38% on Tier 4 — a remarkable pace of improvement, but substantial headroom remains. In January 2026, Epoch AI added an Open Problems component containing unsolved research mathematics questions; none have been solved by AI.

ARC-AGI

ARC-AGI (Abstraction and Reasoning Corpus) tests something fundamentally different from knowledge benchmarks: the ability to infer abstract visual patterns from just a few examples. Each task shows 2-3 input-output grid pairs, and the model must predict the output for a new input — requiring genuine abstraction rather than pattern matching from training data. ARC-AGI-2 (2025) saw top Kaggle scores of only 24% despite 1,455 teams competing. ARC-AGI-3 (released early 2026) raises the stakes further by requiring interactive reasoning — exploration, planning, memory, and goal acquisition. Humans solve 100% of ARC-AGI-3 environments; frontier AI systems score below 1%. This benchmark represents the clearest gap between current LLMs and human-like general intelligence.

Code and Software Engineering Benchmarks

Code evaluation is uniquely tractable: unlike open-ended text, code can be tested against automated test suites. A function either passes its tests or it does not. This has made code benchmarks among the most reliable — and among the most gamed.

HumanEval and MBPP: historical baselines

HumanEval (OpenAI, 2021) provides 164 Python programming problems with unit tests. MBPP (Google, 2021) adds 974 crowd-sourced problems. Both benchmarks now have documented training data contamination: their problems have appeared in countless code-training corpora, making scores unreliable for comparing frontier models. They remain useful only as a very rough filter for basic code generation ability.

SWE-bench Verified

SWE-bench (Princeton NLP) tests a far more realistic task: given a real GitHub issue description, produce a code patch that passes the repository's test suite. The Verified subset filters for high-quality, unambiguous issues. This is closer to actual software engineering — the model must navigate large codebases, understand context, and produce patches that integrate correctly. As of early 2026, top scores hover around 81%. SWE-bench Pro, a harder variant with more complex multi-file issues, shows best-model scores of only ~23%, revealing how much headroom remains for real-world software engineering.

LiveCodeBench

LiveCodeBench (ICLR 2025 Spotlight) continuously collects new competitive programming problems from LeetCode, AtCoder, and Codeforces. Its key innovation is temporal segmentation: problems are dated, so evaluators can test only on problems released after a model's training cutoff. This has proven effective at detecting contamination — LiveCodeBench analysis confirmed contamination artifacts in GPT-4o, Claude, DeepSeek, and Codestral, where performance on pre-cutoff problems significantly exceeds post-cutoff problems.

MLE-bench

MLE-bench (OpenAI, ICLR 2025 Oral) evaluates a different dimension of coding ability: ML engineering. It presents 75 Kaggle machine learning competitions — training models, preparing datasets, running experiments, optimizing submissions. The best setup (o1-preview with AIDE scaffolding) achieves Kaggle bronze medal level in 16.9% of competitions, reaching 34.1% with 8 attempts. This benchmark tests not just code generation but experimental design and iterative refinement.

The pass@k metric

Code benchmarks typically report pass@k: the probability that at least one of $k$ generated solutions passes all tests. The naive estimator (generate $k$ samples, check if any passes) has high variance. The standard unbiased estimator, used across HumanEval, SWE-bench, and LiveCodeBench, computes pass@k from $n$ total samples where $c$ are correct:

$$\text{pass@}k = 1 - \frac{\binom{n-c}{k}}{\binom{n}{k}}$$

Here $\binom{n-c}{k}$ counts the ways to choose $k$ samples that are all incorrect (a combinatorial count, yielding a scalar), and $\binom{n}{k}$ counts all ways to choose $k$ from $n$. Their ratio is the probability that all $k$ chosen samples are wrong; subtracting from 1 gives the probability that at least one is correct.

The following implementation computes this metric numerically.


import torch
from math import comb

def pass_at_k(n: int, c: int, k: int) -> float:
    """
    Unbiased estimator for pass@k.

    Args:
        n: total number of generated samples per problem
        c: number of correct samples (passing all tests)
        k: the k in pass@k (how many attempts the user gets)
    Returns:
        Probability that at least one of k samples is correct.
    """
    if n - c < k:
        # Fewer wrong answers than k — guaranteed to include a correct one
        return 1.0
    return 1.0 - comb(n - c, k) / comb(n, k)

def pass_at_k_batch(n_samples: int, num_correct: torch.Tensor, k: int) -> float:
    """
    Average pass@k across a batch of problems.

    Args:
        n_samples: number of samples generated per problem (same for all)
        num_correct: (num_problems,) — count of passing samples per problem
        k: the k in pass@k
    Returns:
        Mean pass@k across all problems.
    """
    scores = [pass_at_k(n_samples, c.item(), k) for c in num_correct]
    return sum(scores) / len(scores)

# Example: 5 problems, 20 samples each, varying correctness
num_correct = torch.tensor([1, 5, 0, 12, 3])
for k in [1, 5, 10]:
    score = pass_at_k_batch(n_samples=20, num_correct=num_correct, k=k)
    print(f"pass@{k}: {score:.3f}")
# pass@1: 0.210, pass@5: 0.662, pass@10: 0.879

Open-Ended Generation and Human Preference

Multiple-choice benchmarks test knowledge and reasoning, but most real-world uses of language models involve open-ended generation: writing, conversation, instruction following, creative tasks. Evaluating these requires fundamentally different methods — there is no single correct answer to "write me a marketing email" or "explain quantum computing to a 10-year-old."

MT-Bench and AlpacaEval

MT-Bench provides 80 multi-turn questions across 8 categories (writing, roleplay, coding, math, etc.) and uses an LLM judge (originally GPT-4) to score responses on a 1-10 scale. Its multi-turn design captures something single-turn benchmarks miss: whether a model can maintain coherence and follow up on previous context. AlpacaEval tests single-turn instruction following with a simpler metric: win rate against a reference model (originally GPT-4). AlpacaEval 2.0 introduced length-controlled win rates to penalize models that gain advantage through verbosity rather than quality.

Chatbot Arena / LMArena

The most influential open-ended evaluation is Chatbot Arena, rebranded as LMArena in January 2026. Users submit any prompt; two anonymous models generate responses; the user votes for the better one. Votes are aggregated into Elo ratings using the Bradley-Terry model. The system has collected millions of votes across thousands of users, making it the closest approximation to real-world model preference that exists.

The Arena has expanded into domain-specific variants: SciArena (Allen AI, 2025) focuses on scientific literature tasks with votes from 20,000+ researchers, and BioMedArena (LMArena + NIH/DataTecnica, 2025) targets biomedical evaluation. These domain arenas often produce different rankings than the general arena — a model that excels at casual conversation may underperform on scientific reasoning.

WildBench and GDPval

WildBench (Allen AI, ICLR 2025) extracts 1,024 tasks from real user-chatbot conversations and uses task-specific checklists with LLM-as-judge evaluation. Its WB-Reward score achieves a Pearson correlation of 0.98 with top-ranking models on Chatbot Arena — making it a highly reliable automated proxy for human preference that can be run without collecting live user votes.

GDPval (OpenAI, October 2025) measures something more ambitious: how well models perform on real professional work. It presents 1,320 tasks across 44 occupations from the top 9 industries contributing to U.S. GDP — legal briefs, engineering blueprints, nursing care plans, financial analysis. Expert evaluators compare AI vs. human output. The benchmark found that frontier models perform certain professional tasks roughly 100x faster at a fraction of the cost, though quality varies significantly by domain.

IFEval: instruction following

IFEval provides 500 prompts with 25 types of verifiable constraints: "write in more than 400 words," "mention keyword X at least 3 times," "respond entirely in Spanish." Unlike subjective quality judgments, these constraints are unambiguously checkable. IFEval has expanded into multilingual (M-IFEval: French, Japanese, Spanish), function calling (IFEval-FC), and speech variants, tracking how well instruction following transfers across modalities.

HELM

Stanford's HELM (Holistic Evaluation of Language Models) takes the most comprehensive approach: it evaluates models across dozens of scenarios, multiple metrics per scenario (accuracy, calibration, robustness, fairness, bias, toxicity, efficiency), and publishes all results in a standardized, reproducible format. HELM is less of a single benchmark and more of a meta-evaluation framework that aggregates many benchmarks under consistent methodology.

LLM-as-Judge

Human evaluation is the gold standard for open-ended generation quality, but it is slow, expensive, and noisy. The pragmatic alternative — using one language model to evaluate another — has become the default approach for most open-ended benchmarks. MT-Bench, AlpacaEval, WildBench, and countless internal evaluation pipelines all rely on LLM judges. Understanding the methodology, biases, and recent improvements is essential for interpreting any result from these systems.

Core methodology

There are three main paradigms. Single-answer grading asks the judge to score one response on a rubric (e.g., 1-10). Pairwise comparison presents two responses and asks which is better — this is more reliable because relative judgments are easier than absolute ones. Reference-based grading provides a gold-standard answer and asks the judge to score the candidate against it. Pairwise comparison is the most widely used because it sidesteps the calibration problem of absolute scores.

Known biases

LLM judges have systematic biases that must be accounted for. Verbosity bias: judges tend to prefer longer responses even when shorter ones are more accurate — AlpacaEval 2.0 introduced length-controlled win rates specifically to counteract this. Self-preference bias: GPT-4 as a judge tends to rate GPT-4 outputs higher than equivalent outputs from other models. Position bias: the judge may prefer whichever response appears first (or last) in the prompt. Standard mitigations include swapping the order of responses and averaging, using multiple different judge models, and explicitly prompting the judge to ignore length.

Recent improvements

The LLM-as-judge methodology has matured rapidly since 2024. Chain-of-thought judging improves correlation with human judgments from 0.51 to 0.66 (Spearman $\rho$ on summarization tasks) by asking the judge to reason before scoring. Multi-agent debate (MAJ-Eval) automatically extracts personas and conducts group deliberation over outputs, achieving higher alignment with human ratings than single-judge approaches. Crowd Comparative Reasoning uses diverse LLM responses as a "synthetic crowd" for chain-of-thought critiques, improving accuracy by 6.7 percentage points. Mixture of Prompts (MoPs) dynamically selects specialized prompt modules based on input characteristics rather than using a one-size-fits-all judge prompt.

Despite these improvements, fundamental challenges remain. LLM judges cannot reliably evaluate factual correctness on topics outside their training data, they struggle with domain-specific quality (medical, legal, scientific), and they sometimes disagree with human experts in ways that are difficult to detect without human oversight. The best current practice is to combine automated LLM-as-judge with periodic human evaluation as a calibration check.

Agentic Evaluation

A language model that can answer questions is useful. A language model that can act — browse the web, call APIs, write and execute code, manage files, interact with databases — is transformative. But evaluating agents is fundamentally harder than evaluating chatbots: you must check not just the final answer but the entire trajectory of actions, and the environment may change state in response to those actions.

Why agentic evaluation is different

Standard benchmarks present a prompt and grade a response. Agent benchmarks present a goal and grade the outcome after the agent has taken multiple steps through an environment. This introduces several new dimensions: does the agent choose the right tools? Does it recover from errors? Does it avoid destructive side effects? Is it efficient, or does it take 50 steps where 5 would suffice?

GAIA and GAIA-2

GAIA (Meta + HuggingFace) presents 466 questions requiring multi-step reasoning, web browsing, tool use, and multimodality across three difficulty levels. The original GAIA was substantially solved by late 2025, with top models reaching ~90%. GAIA-2 raises the bar with dynamic, asynchronous environments that evolve independently of the agent's actions — the agent must handle temporal constraints and adapt to changing events. Best scores on GAIA-2 hover around 42%.

WebArena

WebArena (CMU) tests autonomous web task completion across real web applications: e-commerce sites, social forums, code repositories, and content management systems. The agent must navigate, click, fill forms, and execute multi-page workflows. Success rates have climbed from 14% at launch to about 60% by early 2026, with IBM's CUGA agent setting a record at ~62%. The gap between "can navigate a single page" and "can complete a multi-step workflow reliably" remains substantial.

Tau-bench

Tau-bench (Sierra Research, ICLR 2025) evaluates customer service agents across airline, retail, and telecom domains. Uniquely, it checks both the final answer and the resulting database state changes — did the agent correctly modify the booking, update the inventory, process the return? GPT-4o succeeds on less than 50% of tasks. More revealingly, consistency (pass@8) drops below 25% in retail, meaning that even when a model can sometimes complete a task, it cannot do so reliably. This gap between demo-quality and production-quality agent performance is the central challenge for deployment.

BFCL V4: function calling

The Berkeley Function Calling Leaderboard (UC Berkeley Gorilla project) evaluates a more atomic skill: can the model correctly format and execute function calls? Version 4 (2025) tests serial and parallel function calls across multiple programming languages, with new categories for web search, memory, and format sensitivity. Frontier models excel at single-turn calls but struggle with memory, dynamic decision-making, and long-horizon reasoning across many sequential function calls.

Evaluating Reasoning Models

The release of OpenAI's o1 in September 2024 created a new category of model — one that spends variable compute "thinking" before answering. DeepSeek R1, o3, o4-mini, and Claude's extended thinking mode followed. These models perform dramatically better on hard reasoning tasks, but they also break assumptions that many evaluation protocols were built on.

Why standard benchmarks undercount reasoning ability

Many benchmarks were designed for models that produce answers in a single forward pass. Reasoning models that allocate additional thinking tokens can solve problems that standard models cannot — AIME 2025 jumps from ~60% for standard models to 100% for the best reasoning model. But this extra compute is not free: a reasoning model may use 10-100x more tokens (and therefore cost and latency) to answer. Simply comparing accuracy without accounting for compute is misleading — it is like comparing a student who gets 5 minutes per question to one who gets an hour.

Compute-optimal evaluation

The emerging best practice is compute-normalized comparison: compare models at the same cost (or same token budget) rather than at arbitrary settings. This means plotting accuracy as a function of compute (tokens or dollars) and comparing the curves. A reasoning model that achieves 95% at $0.15 per question might be inferior to a standard model that achieves 90% at $0.003 — depending on the use case. Benchmarks like AIME and FrontierMath are beginning to report both raw accuracy and cost-normalized accuracy.

Chain-of-thought monitorability

An important 2025 finding from OpenAI: monitoring the hidden chain of thought (the internal reasoning trace) for misbehavior is far more effective than monitoring only the final actions and outputs. When reasoning models reason about deception, reward hacking, or policy circumvention in their chain of thought, this can be detected — but only if the chain of thought is inspectable. This has implications for evaluation: safety benchmarks for reasoning models should evaluate not just what the model does but what it thinks about doing. The tension between monitorability and the model's incentive to obscure its reasoning is an active area of alignment research.

Key Insight

Reasoning models break the assumption that "more compute = better model." They introduce a new axis: how efficiently does the model use its thinking budget? Two models with the same accuracy on FrontierMath may differ by 10x in token cost. The evaluation frontier is no longer one-dimensional.

Safety and Alignment Evaluation

A model that scores well on knowledge benchmarks but generates toxic content, fabricates facts, or exhibits social biases is not deployable. Safety evaluation has evolved from single-metric toxicity scores to comprehensive frameworks that test dozens of harm categories under diverse attack strategies.

Hallucination benchmarks

TruthfulQA presents 817 questions that humans commonly answer incorrectly — questions where the model is likely to reproduce popular misconceptions. It remains the standard hallucination benchmark, though SimpleQA (discussed in Section 2) provides a complementary measure focused on factual accuracy on straightforward questions.

Bias and toxicity

BBQ (Bias Benchmark for QA) tests bias across nine social dimensions — age, disability, gender, nationality, physical appearance, race, religion, sexual orientation, and socioeconomic status — using ambiguous and disambiguated question pairs. ToxiGen generates implicit hate speech to test whether models can both detect and avoid producing toxic content across 13 demographic groups. The Perspective API provides a continuous toxicity score (0-1) that can be applied to any model output.

HarmBench

HarmBench standardizes red-teaming evaluation with a key innovation: it provides Attack Success Rate (ASR) classifiers that automate the scoring of whether a model's response constitutes a harmful completion. Rather than relying on human judges to decide "did the model comply with this harmful request?", HarmBench trains classifiers to make this determination reproducibly. It covers a broad taxonomy of harmful behaviors and provides standard train/test splits, enabling fair comparison of both attack methods and defense mechanisms.

ALERT

ALERT (Babelscape, 2024) provides 45,000+ instructions across a fine-grained taxonomy of 6 macro categories and 32 micro safety risk categories. The extended version adds 30,000+ adversarial prompts with roughly 7,000 per attack strategy (jailbreaking, prompt injection, role-playing, etc.). This granularity matters: a model might be perfectly safe against direct harmful requests but vulnerable to role-playing attacks in a specific harm category. ALERT exposes these failure patterns.

Red-teaming protocols

Red-teaming — structured adversarial evaluation by humans or AI systems — has evolved from ad-hoc prompt engineering into standardized protocols. RedBench (2026) provides a universal dataset for comprehensive red teaming. Domain-specific variants have appeared: FINBench assesses regulatory risk concealment in financial LLMs, and SGToxicGuard tests safety in Singapore's multilingual context (Singlish, Chinese, Malay, Tamil). The trend is clear: generic safety benchmarks are giving way to domain-specific and culturally-specific evaluations that capture failure modes invisible to English-centric, general-purpose benchmarks.

Constitutional AI self-critique scoring

Anthropic's Constitutional AI (CAI) introduces a self-evaluation loop: the model generates a response, then critiques its own output against a set of written principles (the "constitution"), then revises. This can be scored by measuring how often the model's self-critique correctly identifies problematic content, how much the revised response improves, and whether the final output satisfies the constitutional principles. CAI self-critique scoring provides an automated, scalable measure of a model's alignment — essentially asking "does the model know when it is being harmful?" The critique accuracy (how often the model catches its own violations) and revision quality (how much harm is reduced after self-revision) together give a two-dimensional safety metric that complements external red-teaming evaluations.

Contamination and Evaluation Validity

Every benchmark discussed so far is vulnerable to the same fundamental threat: if the test questions appear in the model's training data, the benchmark measures memorization rather than capability. This is not a theoretical concern — contamination has been documented in multiple major benchmarks and has distorted leaderboard rankings.

The contamination problem

Modern language models train on trillions of tokens scraped from the internet. Benchmark datasets — MMLU questions, HumanEval code problems, TruthfulQA items — live on the internet. Unless extreme care is taken to exclude them, benchmarks end up in pre-training corpora. The result: inflated scores that reflect data memorization rather than genuine reasoning. HumanEval is now known to have widespread contamination, making pre-2024 code generation comparisons unreliable.

Detection methods

N-gram overlap is the simplest approach: check if long sequences from benchmark items appear verbatim in the training corpus. This catches direct contamination but misses paraphrased or reformatted versions. Perplexity analysis is more subtle: if a model has unexpectedly low perplexity on benchmark items compared to similar items not in the benchmark, contamination is likely. LiveCodeBench uses temporal segmentation as both a detection and prevention mechanism: by testing only on problems released after a model's training cutoff, contaminated items are excluded by construction.

Dynamic benchmarks

LiveBench (ICLR 2025 Spotlight) is the most ambitious dynamic benchmark: it updates monthly with questions drawn from recent arXiv papers, news articles, and IMDb synopses. Every question has an objective, automatically verifiable ground-truth answer — no LLM judge needed. It covers math, coding, reasoning, language, instruction following, and data analysis. Top models still achieve below 70%, and the continuous updates ensure that no model can benefit from memorizing past versions.

LiveCodeBench applies the same philosophy to code: it continuously collects new competitive programming problems with temporal metadata. Its time-segmented evaluation has confirmed contamination patterns in multiple major models, where performance drops sharply on problems released after the model's training cutoff.

EvoEval takes a complementary approach: instead of finding new problems, it algorithmically transforms existing benchmark problems — rephrasing, combining, decomposing, or inverting them — to generate novel variants that test the same skills but cannot be memorized. This lets researchers "resurrect" saturated benchmarks by generating fresh test instances from their structure, providing a middle ground between fully static and fully dynamic evaluation.

Long-context evaluation: HELMET

A separate validity concern affects long-context evaluation. HELMET (ICLR 2025) tested models on long-context tasks up to 128K tokens across seven application categories and discovered that popular synthetic tasks like needle-in-a-haystack (finding a specific fact buried in irrelevant text) do not reliably predict downstream long-context performance. A model that achieves 100% on needle-in-a-haystack may still fail at long-document summarization, multi-document QA, or citation-grounded generation. HELMET recommends evaluating on realistic long-context applications rather than synthetic probes, and found that open-source models significantly lag behind closed models on genuine long-context reasoning.

Reproducibility

Even without contamination, evaluation results vary significantly based on seemingly minor choices: temperature (greedy vs. sampled decoding), system prompt (different preambles can swing scores by 5+ percentage points), few-shot format (how examples are presented), and decoding strategy (beam search vs. nucleus sampling). The field is moving toward standardized evaluation harnesses — notably lm-evaluation-harness from EleutherAI — that fix these choices and enable reproducible comparisons. Any reported benchmark score that does not specify these parameters should be treated with skepticism.

Warning

A model's benchmark score is only meaningful if you know: (1) which exact version of the benchmark was used, (2) whether contamination was checked, (3) what inference parameters were used, and (4) when the evaluation was run relative to the model's training cutoff. Leaderboard rankings without this metadata are marketing, not science.

Implementing Elo Ratings for Model Comparison

The Elo rating system — originally designed for chess — underlies Chatbot Arena's leaderboard. Understanding its mechanics clarifies both its power and its limitations. The core idea: each model has a rating; when two models are compared, the expected outcome is a function of the rating difference; the actual outcome updates both ratings.

The expected score for model A against model B is computed from their rating difference:

$$E_A = \frac{1}{1 + 10^{(R_B - R_A)/400}}$$

Here $R_A$ and $R_B$ are the current Elo ratings (scalars). The constant 400 controls the scale: a 400-point rating difference corresponds to a 10:1 expected win ratio. After a match, ratings update as $R'_A = R_A + K \cdot (S_A - E_A)$ where $S_A$ is the actual outcome (1 for win, 0.5 for tie, 0 for loss) and $K$ is the update magnitude (typically 4-32).

The following code implements a complete Elo rating system suitable for evaluating model comparisons.


import torch

class EloRatingSystem:
    """
    Elo rating system for pairwise model comparison,
    as used in Chatbot Arena / LMArena.
    """
    def __init__(self, k: float = 32.0, initial_rating: float = 1500.0):
        self.k = k                   # update magnitude (scalar)
        self.initial_rating = initial_rating
        self.ratings = {}            # model_name -> current Elo rating

    def expected_score(self, rating_a: float, rating_b: float) -> float:
        """P(A wins) based on current ratings. Returns scalar in [0, 1]."""
        return 1.0 / (1.0 + 10.0 ** ((rating_b - rating_a) / 400.0))

    def update(self, model_a: str, model_b: str, outcome: float):
        """
        Update ratings after a pairwise comparison.

        Args:
            model_a: name of first model
            model_b: name of second model
            outcome: 1.0 = A wins, 0.0 = B wins, 0.5 = tie
        """
        ra = self.ratings.setdefault(model_a, self.initial_rating)
        rb = self.ratings.setdefault(model_b, self.initial_rating)

        ea = self.expected_score(ra, rb)  # scalar: expected score for A
        eb = 1.0 - ea                      # scalar: expected score for B

        self.ratings[model_a] = ra + self.k * (outcome - ea)
        self.ratings[model_b] = rb + self.k * ((1.0 - outcome) - eb)

    def get_leaderboard(self):
        """Return models sorted by Elo rating, descending."""
        return sorted(self.ratings.items(), key=lambda x: -x[1])

# Simulate a small arena with 4 models and 100 comparisons
elo = EloRatingSystem(k=16.0)
torch.manual_seed(42)

models = ["ModelA", "ModelB", "ModelC", "ModelD"]
# True skill levels (hidden): A > B > C > D
true_skill = {"ModelA": 0.8, "ModelB": 0.6, "ModelC": 0.4, "ModelD": 0.2}

for _ in range(200):
    # Random matchup
    i, j = torch.randint(0, 4, (2,)).tolist()
    while i == j:
        j = torch.randint(0, 4, (1,)).item()
    a, b = models[i], models[j]

    # Simulate outcome based on skill difference
    p_a_wins = torch.sigmoid(torch.tensor(
        5.0 * (true_skill[a] - true_skill[b])  # scalar logit
    )).item()
    outcome = 1.0 if torch.rand(1).item() < p_a_wins else 0.0
    elo.update(a, b, outcome)

print("Leaderboard after 200 comparisons:")
for name, rating in elo.get_leaderboard():
    print(f"  {name}: {rating:.0f}")
# Rankings should recover true skill ordering: A > B > C > D

Chatbot Arena uses a more sophisticated variant: the Bradley-Terry model, which jointly estimates all ratings from the full matrix of pairwise comparisons using maximum likelihood rather than sequential updates. This gives more stable ratings but follows the same core principle — pairwise comparisons become global rankings through probabilistic modeling of relative strength.
