Chapter 07

Encoder-Only Models — BERT and Beyond

Before BERT, every NLP task was its own engineering project: a custom architecture, custom features, custom training pipeline. A sentiment classifier shared nothing with a question answering system. Then in October 2018, a single pre-trained Transformer encoder shattered records on eleven tasks simultaneously — with nothing more than a thin classification head swapped on top. The idea was shockingly simple: train a deep bidirectional encoder to fill in blanks, then repurpose its internal representations for any downstream task. By the end of this chapter you will understand how BERT's masked language model works, why bidirectional context gives it decisive advantages on understanding tasks, how a half-dozen subsequent variants improved the recipe, and when encoder-only models remain the right architectural choice even in the era of massive generative models.

BERT (Devlin et al., 2018)

The acronym stands for Bidirectional Encoder Representations from Transformers, and the key word is bidirectional. GPT, which appeared four months earlier, used a left-to-right causal mask: each token could only attend to tokens before it. This is natural for generation — you predict the next word from the preceding words — but it is a severe handicap for understanding tasks. When deciding whether "bank" means a financial institution or a river bank, you need context on both sides. BERT removed the causal mask entirely, letting every token attend to every other token in the sequence.

Architecture

BERT is a stack of standard Transformer encoder layers with no decoder. Two sizes were released:

ModelLayersHidden size ($d_\text{model}$)Attention headsParameters
BERT-Base1276812110M
BERT-Large24102416340M

The input to BERT is the sum of three embeddings: a token embedding, a segment embedding (marking whether a token belongs to sentence A or sentence B), and a learned positional embedding. The maximum sequence length is 512 tokens. A special [CLS] token is prepended, and its final-layer representation serves as the aggregate sequence representation for classification tasks. A [SEP] token separates the two segments.

Pre-training Task 1: Masked Language Modeling (MLM)

The central pre-training objective is the masked language model. Before feeding a sentence to the encoder, 15% of its tokens are selected for prediction. Of those selected tokens, 80% are replaced with a special [MASK] token, 10% are replaced with a random token from the vocabulary, and 10% are left unchanged. The model must predict the original token at every selected position.

This 80/10/10 split is not arbitrary. If all selected tokens were replaced with [MASK], the model would learn representations that are useful only when [MASK] is present — a token that never appears during fine-tuning. The random replacement and unchanged tokens force the model to maintain good representations for all positions, since any position might need to be predicted. The model can never be sure which tokens are "real" and which are corrupted, so it must build strong contextual representations everywhere.

The MLM loss is standard cross-entropy, but only over the selected positions:

$$\mathcal{L}_\text{MLM} = -\sum_{i \in \mathcal{M}} \log P(x_i \mid \tilde{x})$$

Here $\mathcal{M}$ is the set of masked positions and $\tilde{x}$ is the corrupted input. Because only 15% of tokens contribute to the loss, BERT is less sample-efficient than autoregressive models (which get a loss signal from every token). This is a well-known tradeoff: bidirectionality improves representation quality but reduces training efficiency per token. BERT compensates by training on large corpora for many steps — the original model trained on BooksCorpus (800M words) and English Wikipedia (2,500M words) for 1M steps with a batch size of 256.

Here is a minimal PyTorch implementation of the MLM head and the 80/10/10 masking strategy:


import torch
import torch.nn as nn
import torch.nn.functional as F

class MLMHead(nn.Module):
    """Predict original tokens at masked positions."""
    def __init__(self, d_model, vocab_size):
        super().__init__()
        self.dense = nn.Linear(d_model, d_model)
        self.layer_norm = nn.LayerNorm(d_model)
        self.projection = nn.Linear(d_model, vocab_size)

    def forward(self, hidden_states, masked_positions):
        # Extract representations at masked positions only
        masked_hidden = hidden_states[masked_positions]       # (num_masked, d_model)
        x = self.layer_norm(F.gelu(self.dense(masked_hidden)))
        logits = self.projection(x)                           # (num_masked, vocab_size)
        return logits

def create_mlm_batch(token_ids, vocab_size, mask_token_id=103, mask_prob=0.15):
    """Apply BERT's masking strategy: 80% [MASK], 10% random, 10% keep."""
    labels = token_ids.clone()
    # Select 15% of positions to predict
    mask = torch.rand(token_ids.shape) < mask_prob
    labels[~mask] = -100  # only compute loss on masked positions

    # Of the selected positions: 80% become [MASK]
    indices = mask.nonzero(as_tuple=True)
    rand = torch.rand(indices[0].shape)
    token_ids[indices[0][rand < 0.8], indices[1][rand < 0.8]] = mask_token_id
    # 10% become a random token
    random_mask = (rand >= 0.8) & (rand < 0.9)
    token_ids[indices[0][random_mask], indices[1][random_mask]] = torch.randint(0, vocab_size, (random_mask.sum(),))
    # 10% stay unchanged — model must still predict them
    return token_ids, labels

Pre-training Task 2: Next Sentence Prediction (NSP)

The second objective, Next Sentence Prediction, trains the model to predict whether two segments are consecutive in the original document. Half the training pairs are genuine consecutive sentences (labeled IsNext); the other half pair a sentence with a random sentence from a different document (labeled NotNext). The prediction is made from the [CLS] token's representation via a binary classifier.

NSP was intended to improve performance on tasks requiring reasoning about sentence pairs, like natural language inference and question answering. In retrospect, it turned out to be BERT's weakest design choice. RoBERTa later showed that removing NSP entirely and training only with MLM — using full-document inputs rather than sentence pairs — significantly improves downstream performance. The NSP task was too easy: the model could often distinguish unrelated sentences based on topic mismatch alone, without learning genuine inter-sentence reasoning.

Fine-tuning Paradigm

BERT's fine-tuning recipe is elegant in its simplicity: take the pre-trained encoder, add a thin task-specific head on top, and fine-tune all parameters end-to-end on the target task's labeled data. The task head is typically a single linear layer.

For classification tasks (sentiment, NLI), the [CLS] representation is projected through a linear layer to $C$ classes: $\hat{y} = \text{softmax}(W\,h_{\text{[CLS]}} + b)$ (matrix multiply, $\mathbb{R}^{C \times d} \times \mathbb{R}^{d \times 1} \to \mathbb{R}^{C \times 1}$). For span extraction (SQuAD), two linear layers predict the start and end positions of the answer span. For token classification (NER), each token's final representation is projected to the tag set. In every case, fine-tuning typically takes 2–4 epochs with a small learning rate ($2 \times 10^{-5}$ to $5 \times 10^{-5}$) and is finished in under an hour on a single GPU.

Here is a minimal PyTorch implementation of a BERT classification head and fine-tuning loop:


class BERTClassifier(nn.Module):
    """Fine-tune BERT for sequence classification using the [CLS] token."""
    def __init__(self, bert_encoder, d_model, num_classes, dropout=0.1):
        super().__init__()
        self.encoder = bert_encoder           # pre-trained Transformer encoder
        self.dropout = nn.Dropout(dropout)
        self.classifier = nn.Linear(d_model, num_classes)

    def forward(self, input_ids, attention_mask):
        # Run full encoder: (batch, seq_len, d_model)
        hidden_states = self.encoder(input_ids, attention_mask)
        # Extract [CLS] token representation (position 0)
        cls_output = hidden_states[:, 0, :]   # (batch, d_model)
        cls_output = self.dropout(cls_output)
        logits = self.classifier(cls_output)  # (batch, num_classes)
        return logits

Contextualized Representations

Before BERT, word representations were static: word2vec (Mikolov et al., 2013) and GloVe (Pennington et al., 2014) assigned a single vector to each word regardless of context. The word "bank" had one embedding whether it appeared in "river bank" or "investment bank." ELMo (Peters et al., 2018) partially addressed this by concatenating forward and backward LSTM hidden states, but the two directions were independently computed and merely concatenated — not jointly attended.

BERT's representations are deeply contextualized: the same word produces different vectors in different contexts because every layer of the encoder integrates information from every other token via bidirectional self-attention. Probing experiments revealed that BERT's representations encode a remarkably rich hierarchy of linguistic structure: lower layers capture surface-level features (part-of-speech), middle layers capture syntactic structure (dependency parse), and upper layers capture task-specific semantic information.

BERT's Impact

When BERT was released, it achieved state-of-the-art results on all eleven tasks in the GLUE benchmark, the SQuAD 1.1 and 2.0 question answering benchmarks, and the SWAG commonsense reasoning benchmark. In several cases it surpassed human performance. The results were not marginal — BERT-Large improved the GLUE score by 7.7 absolute points over the previous best.

The deeper impact was methodological. Before BERT, NLP practitioners spent most of their effort on feature engineering and architecture design for specific tasks. After BERT, the standard workflow became: take the pre-trained model, fine-tune it, ship it. The field consolidated around a single architecture and a single pre-training recipe, and the research focus shifted from task-specific innovations to better pre-training, better data, and larger scale. This paradigm shift — pre-train once, fine-tune everywhere — remains the foundation of modern NLP.

BERT Variants and Improvements

Within a year of BERT's release, a series of follow-up works systematically improved every aspect of the recipe: the pre-training objective, the data regime, the attention mechanism, the model size, and the training efficiency. Each of these variants isolated a specific weakness in the original BERT and fixed it, collectively advancing encoder-only models far beyond the original baseline.

RoBERTa: The Recipe Matters More Than the Architecture

Liu et al. (2019) published RoBERTa (Robustly Optimized BERT Approach) with a simple but devastating thesis: BERT was significantly undertrained, and most of the gap to optimal performance could be closed by fixing the training recipe without changing the architecture at all.

The changes were:

  • Remove NSP entirely. Train with MLM only, using full-document inputs packed to the 512-token limit. This improved downstream performance on all tasks.
  • Dynamic masking. BERT applied masking once during data preprocessing, so every epoch saw the same masks. RoBERTa generates fresh random masks for each input on every pass through the data, effectively multiplying the diversity of training signals.
  • More data, longer training. RoBERTa trained on 160GB of text (10× BERT's 16GB) including CC-News, OpenWebText, and Stories. Training ran for 500K steps with a larger batch size of 8,192 sequences.
  • Larger batches. Increasing the batch size from 256 to 8,192 improved optimization stability and MLM accuracy.

The result: RoBERTa matched or exceeded XLNet (which introduced a complex permutation-based objective) on every benchmark, proving that the original Transformer encoder architecture was not the bottleneck — the training recipe was. This lesson recurs throughout deep learning: the architecture often matters less than the data, compute, and hyperparameter choices.

ALBERT: Smaller, But Not Weaker

Lan et al. (2019) introduced ALBERT (A Lite BERT) with two parameter-reduction techniques. Factorized embedding parameterization decouples the vocabulary embedding size $E$ from the hidden size $H$: instead of a single $V \times H$ embedding matrix (which is huge when $H = 4096$), ALBERT uses a $V \times E$ matrix followed by an $E \times H$ projection, with $E \ll H$. Cross-layer parameter sharing reuses the same Transformer block weights across all layers. An ALBERT-xxlarge model with 12M unique parameters (vs BERT-Large's 340M) actually outperformed BERT-Large on several benchmarks.

ALBERT also replaced NSP with a Sentence Order Prediction (SOP) task, where the model must distinguish correctly ordered sentence pairs from swapped ones. SOP is harder than NSP (both sentences come from the same document, so topic cues are absent) and consistently improved downstream performance on multi-sentence tasks.

DeBERTa: Disentangled Attention

He et al. (2020) proposed DeBERTa (Decoding-enhanced BERT with Disentangled Attention), which improved BERT by separating the roles of content and position in the attention computation. In standard BERT, each token's representation is the sum of its content embedding and its position embedding: $H_i = E_{\text{token}}(x_i) + E_{\text{pos}}(i)$. This means the attention score $H_i^\top H_j$ (dot product, yields a scalar) conflates four distinct interactions: content-to-content, content-to-position, position-to-content, and position-to-position.

DeBERTa computes the attention score as the sum of three disentangled terms:

$$A_{ij} = \underbrace{H_i^c {H_j^c}^\top}_{\text{content-to-content}} + \underbrace{H_i^c {P_{i|j}}^\top}_{\text{content-to-position}} + \underbrace{P_{j|i} {H_j^c}^\top}_{\text{position-to-content}}$$

Here $H^c$ denotes content representations and $P$ denotes relative position embeddings. The position-to-position term is dropped as empirically uninformative. DeBERTa also introduced an Enhanced Mask Decoder (EMD) that incorporates absolute position information only in the final prediction layer, after the disentangled attention layers have processed relative positions. DeBERTa-V3 (Large) became one of the strongest encoder models, exceeding human performance on the SuperGLUE benchmark.

DistilBERT: Knowledge Distillation

Sanh et al. (2019) applied knowledge distillation to compress BERT-Base into a smaller model called DistilBERT. The student model has 6 layers (half of BERT-Base's 12) and is trained to mimic three signals from the teacher: the soft probability distribution over the vocabulary (distillation loss), the hidden state representations (cosine embedding loss), and the true MLM labels (standard MLM loss).

The result: DistilBERT retains 97% of BERT's performance on GLUE while being 40% smaller (66M parameters) and 60% faster at inference. This demonstrated that BERT's 12 layers contain significant redundancy — much of the information can be compressed into 6 layers with minimal quality loss. DistilBERT became the default choice for production NLP where latency and memory are constrained.

SpanBERT: Masking Spans, Not Random Tokens

Joshi et al. (2020) proposed SpanBERT, which modifies the masking strategy to mask contiguous spans of tokens rather than random individual tokens. Span lengths are sampled from a geometric distribution with mean 3.8, favoring short spans but occasionally producing long ones. The model predicts each masked token using both its position's representation and the representations of the span boundary tokens (the tokens immediately before and after the masked span).

This Span Boundary Objective (SBO) encourages the boundary tokens to encode a summary of the span's content, which proves particularly useful for tasks that require understanding entities and their relationships: coreference resolution, relation extraction, and extractive question answering. SpanBERT set new records on SQuAD 1.1, SQuAD 2.0, and the OntoNotes coreference task.

Multilingual and Domain-Specific Encoders

BERT's pre-train / fine-tune paradigm transfers naturally across languages and domains. The encoder architecture is agnostic to what text it processes — it just needs enough pre-training data to build good representations. This opened two important research directions: multilingual models that handle many languages with a single model, and domain-specific models that specialize in the vocabulary and patterns of particular fields.

mBERT: Multilingual BERT

Google released mBERT alongside the original BERT as a single model trained on Wikipedia text in 104 languages. The architecture is identical to BERT-Base (12 layers, 768 hidden, 110M parameters), with a larger shared WordPiece vocabulary of 110K tokens covering all languages. No language identifiers or explicit cross-lingual alignment signals were used — the model simply consumed mixed-language text during pre-training.

The surprising result was that mBERT learned cross-lingual representations spontaneously. A model fine-tuned for named entity recognition on English data could be applied directly to French, German, or Chinese text with reasonable accuracy — a capability called zero-shot cross-lingual transfer. Pires et al. (2019) showed that this emerged from the shared subword vocabulary and shared Transformer parameters, which forced different languages to occupy overlapping regions of the embedding space.

The limitation is the curse of multilinguality: with a fixed model capacity, spreading parameters across 104 languages necessarily dilutes per-language quality. Low-resource languages suffer disproportionately because they contribute fewer examples to pre-training. This motivated dedicated multilingual models with more capacity.

XLM-R: Cross-lingual Language Model — RoBERTa

Conneau et al. (2020) trained XLM-R (XLM-RoBERTa) on 2.5TB of CommonCrawl data in 100 languages using the RoBERTa recipe. The key differences from mBERT: 10× more training data, larger model (550M parameters for XLM-R Large), and SentencePiece tokenization with a 250K vocabulary. XLM-R dramatically outperformed mBERT on cross-lingual benchmarks like XNLI and MLQA, particularly for low-resource languages that benefited from the larger data mix.

XLM-R demonstrated that the curse of multilinguality can be mitigated by scaling: with enough capacity and data, a single model can maintain strong per-language quality while enabling powerful cross-lingual transfer. It became the standard backbone for multilingual NLP applications and remains competitive in 2025.

Domain-Specific Encoders

General-purpose encoders are pre-trained on web text and Wikipedia, which means their representations are optimized for general-domain language. Specialized domains — biomedicine, law, finance, clinical medicine — use distinctive vocabulary, syntax, and conventions that are underrepresented in general corpora. Domain-specific pre-training addresses this by continuing (or restarting) BERT pre-training on domain text.

BioBERT (Lee et al., 2020) continued pre-training BERT on PubMed abstracts (4.5B words) and PMC full-text articles (13.5B words). It significantly outperformed general BERT on biomedical named entity recognition, relation extraction, and question answering. The gains came primarily from learning domain-specific tokenization patterns (BERT's WordPiece splits "immunoglobulin" into 4 subwords; BioBERT's representations better capture its meaning as a single concept) and statistical regularities in scientific text.

SciBERT (Beltagy et al., 2019) trained from scratch on 1.14M papers from Semantic Scholar, using a custom scientific vocabulary (SCIVOCAB). It outperformed BERT on tasks from the scientific domain including citation intent classification, relation extraction on SciERC, and sequence labeling on biological entities. The custom vocabulary proved particularly important: general-domain BERT splits many scientific terms into uninformative subwords.

LegalBERT (Chalkidis et al., 2020) continued pre-training on 12GB of diverse legal text from EU, UK, and US jurisdictions. Legal language is distinctive: extremely long sentences, nested conditional clauses, precise technical terminology, and frequent cross-references. LegalBERT improved performance on contract review NLI, court opinion classification, and legal NER.

FinBERT (Araci, 2019) targeted financial text, training on a combination of financial news, earnings call transcripts, and SEC filings. Financial sentiment analysis is challenging because the same words carry domain-specific polarity: "volatile" is neutral in general text but negative in finance; "dividend cut" requires understanding that "cut" applies to a financial instrument. FinBERT achieved a 6-point improvement on the Financial PhraseBank sentiment benchmark.

ClinicalBERT (Alsentzer et al., 2019) adapted BERT to clinical text from the MIMIC-III database of de-identified electronic health records. Clinical notes contain abbreviations, misspellings, domain jargon, and implicit temporal reasoning that differ substantially from any web corpus. ClinicalBERT improved performance on clinical NLI, hospital readmission prediction, and mortality prediction tasks.

The general pattern is clear: continued pre-training on 1–20GB of domain text, starting from a strong general checkpoint, consistently yields 2–8% improvements on domain-specific downstream tasks. The cost is modest (a few GPU-days), and the recipe is reliable. This approach remains standard practice for deploying encoder models in specialized industries.

Use Cases for Encoder-Only Models

In 2025, decoder-only generative models dominate headlines. But encoder-only models remain the better choice for a large class of production NLP tasks. Understanding when to use an encoder is a critical architectural decision.

Text Classification

Sentiment analysis, intent detection, toxicity classification, topic categorization — any task that maps a text to a label. The [CLS] token's final representation is passed through a linear layer, and cross-entropy loss trains the system. BERT-Base handles most classification tasks at sub-10ms latency per example, making it practical for real-time applications. A fine-tuned encoder typically matches or exceeds GPT-4's zero-shot classification accuracy on tasks with clear label definitions, at a fraction of the cost and latency.

Named Entity Recognition

Named Entity Recognition (NER) is a token-level classification task: each token is assigned a tag (Person, Organization, Location, etc.) using a BIO or BIOES tagging scheme. Each token's final-layer representation is projected to the tag space. BERT-based NER models dominate production systems for entity extraction in search engines, medical records, financial documents, and legal contracts.

Question Answering: Span Extraction

In extractive question answering, the input is a question-context pair and the output is a span of text within the context that answers the question. Two linear layers predict the start and end positions of the answer span. The model scores each token as a potential start or end position, and the highest-scoring valid span (start ≤ end, within length limits) is returned. BERT set the template for this approach on SQuAD, and variants like SpanBERT and DeBERTa refined it further.

Semantic Textual Similarity

Given two sentences, produce a score indicating their semantic similarity. Sentence-BERT (Reimers and Gurevych, 2019) fine-tuned BERT with a siamese architecture to produce meaningful sentence embeddings. The [CLS] representation (or mean-pooled token representations) from each sentence are compared using cosine similarity. This enables fast nearest-neighbor search: encode your corpus once, then compare query embeddings against the stored embeddings. Encoder-based sentence embeddings power semantic search, duplicate detection, and clustering at scale.

Here is a minimal PyTorch implementation of mean pooling and cosine similarity for sentence embeddings:


def mean_pool(hidden_states, attention_mask):
    """Mean pooling over token embeddings, ignoring padding tokens."""
    # hidden_states: (batch, seq_len, d_model)
    # attention_mask: (batch, seq_len) — 1 for real tokens, 0 for padding
    mask_expanded = attention_mask.unsqueeze(-1).float()  # (batch, seq_len, 1)
    sum_embeddings = (hidden_states * mask_expanded).sum(dim=1)
    sum_mask = mask_expanded.sum(dim=1).clamp(min=1e-9)
    return sum_embeddings / sum_mask  # (batch, d_model)

# Compute cosine similarity between two sentences
with torch.no_grad():
    h1 = encoder(tokens_a, mask_a)  # (1, seq_len, d_model)
    h2 = encoder(tokens_b, mask_b)
    emb_a = mean_pool(h1, mask_a)   # (1, d_model)
    emb_b = mean_pool(h2, mask_b)
    similarity = F.cosine_similarity(emb_a, emb_b)  # scalar in [-1, 1]

Project: Building a Sentence Transformer from Scratch

The mean-pooling snippet above produces sentence embeddings, but those embeddings are not yet good. A vanilla BERT encoder was never trained to place semantically similar sentences near each other in vector space. Reimers and Gurevsky (2019) showed that BERT's raw [CLS] embeddings actually perform worse than GloVe averages on semantic similarity benchmarks. The fix is Sentence-BERT (SBERT): fine-tune the encoder with a Siamese architecture and a contrastive objective so that the resulting embeddings capture sentence-level meaning.

The practical motivation is speed. A cross-encoder — feeding both sentences as a single pair through BERT — is accurate but requires $O(n^2)$ forward passes to compare $n$ sentences. With 10,000 sentences, that is roughly 50 million forward passes, taking ~65 hours. A sentence transformer encodes each sentence once, stores the embedding, and compares embeddings with cosine similarity. Encoding 10,000 sentences takes seconds; pairwise comparison of the stored vectors takes milliseconds.

In this project we build a sentence transformer end-to-end: we take a pre-trained BERT encoder (already trained with MLM to understand language), wrap it with mean pooling, fine-tune it with a contrastive loss on real NLI data, and use the resulting embeddings for semantic search.

Step 1: Sentence Transformer Model

A sentence transformer starts from a pre-trained encoder — one that already understands language from masked language modeling — and wraps it with mean pooling. We load bert-base-uncased as our backbone. The pre-trained weights give us rich token representations; our job is to teach the model to pool those representations into a single sentence vector that captures meaning.


import torch
import torch.nn as nn
import torch.nn.functional as F
from transformers import AutoModel

class SentenceTransformer(nn.Module):
    def __init__(self, model_name="bert-base-uncased"):
        super().__init__()
        self.encoder = AutoModel.from_pretrained(model_name)
        self.d_model = self.encoder.config.hidden_size  # 768 for bert-base

    def forward(self, input_ids, attention_mask):
        # input_ids: (batch, seq_len)   attention_mask: (batch, seq_len)
        out = self.encoder(input_ids=input_ids, attention_mask=attention_mask)
        hidden = out.last_hidden_state         # (batch, seq_len, 768)
        return self._mean_pool(hidden, attention_mask)

    def _mean_pool(self, hidden, mask):
        # hidden: (batch, seq_len, 768)  mask: (batch, seq_len)
        mask_exp = mask.unsqueeze(-1).float()              # (batch, seq_len, 1)
        summed = (hidden * mask_exp).sum(dim=1)            # (batch, 768)
        counts = mask_exp.sum(dim=1).clamp(min=1e-9)       # (batch, 1)
        return summed / counts                              # (batch, 768)

    @torch.no_grad()
    def encode(self, input_ids, attention_mask):
        """Encode sentences to normalized embeddings for similarity search."""
        embs = self.forward(input_ids, attention_mask)     # (batch, 768)
        return F.normalize(embs, p=2, dim=-1)              # unit vectors

The architecture is intentionally minimal. BERT already contains 12 transformer layers with multi-head attention and feed-forward networks — all the heavy lifting that earlier chapters covered in detail. Our SentenceTransformer adds exactly one thing: mean pooling that collapses the per-token outputs into a single vector per sentence. The real work happens in the training objective.

Step 2: Multiple Negatives Ranking Loss

The key training signal comes from Multiple Negatives Ranking (MNR) loss — also known as InfoNCE. Given a batch of $B$ anchor-positive pairs, we compute a $B \times B$ cosine similarity matrix (matrix multiply of L2-normalized embeddings, $\mathbb{R}^{B \times d} \times \mathbb{R}^{d \times B} \to \mathbb{R}^{B \times B}$). The diagonal entries are the correct pairs; every off-diagonal entry is an in-batch negative. A temperature-scaled cross-entropy loss pushes each anchor toward its true positive and away from all negatives:

$$\mathcal{L}_i = -\log \frac{\exp\!\bigl(\text{sim}(a_i, p_i) / \tau\bigr)}{\sum_{j=1}^{B} \exp\!\bigl(\text{sim}(a_i, p_j) / \tau\bigr)}$$

where $\text{sim}(a_i, p_j) = a_i^\top p_j$ is the cosine similarity (dot product of L2-normalized vectors, yielding a scalar in $[-1, 1]$) and $\tau = 1/20 = 0.05$ is the temperature. Larger batches provide more negatives per anchor, producing a stronger learning signal.


class MNRLoss(nn.Module):
    """Multiple Negatives Ranking Loss with in-batch negatives."""
    def __init__(self, scale=20.0):
        super().__init__()
        self.scale = scale               # 1 / temperature
        self.cross_entropy = nn.CrossEntropyLoss()

    def forward(self, anchors, positives):
        # anchors:   (batch, d_model) — already L2-normalized
        # positives: (batch, d_model) — already L2-normalized
        # Cosine similarity matrix via matrix multiply:
        # (batch, d_model) @ (d_model, batch) -> (batch, batch)
        scores = anchors @ positives.T * self.scale
        # Target: each anchor i should match positive i (diagonal)
        labels = torch.arange(scores.size(0), device=scores.device)
        return self.cross_entropy(scores, labels)

Step 3: Training on NLI Data

We train on entailment pairs from SNLI — each pair consists of a premise and a hypothesis that logically follows from it, giving us natural positive pairs for contrastive learning.


from transformers import AutoTokenizer
from datasets import load_dataset
from torch.utils.data import DataLoader

# HuggingFace for tokenizer + data loading; model fine-tuning is all PyTorch
tokenizer = AutoTokenizer.from_pretrained("bert-base-uncased")
ds = load_dataset("stanfordnlp/snli", split="train")
ds = ds.filter(lambda x: x["label"] == 0)  # keep entailment pairs only

model = SentenceTransformer("bert-base-uncased").cuda()  # loads pre-trained weights
optimizer = torch.optim.AdamW(model.parameters(), lr=2e-5, weight_decay=0.01)
criterion = MNRLoss(scale=20.0)

def collate(batch):
    premises = [ex["premise"] for ex in batch]
    hypotheses = [ex["hypothesis"] for ex in batch]
    enc_p = tokenizer(premises, padding=True, truncation=True,
                      max_length=128, return_tensors="pt")
    enc_h = tokenizer(hypotheses, padding=True, truncation=True,
                      max_length=128, return_tensors="pt")
    return enc_p, enc_h

loader = DataLoader(ds, batch_size=32, shuffle=True, collate_fn=collate)

model.train()
for step, (enc_p, enc_h) in enumerate(loader):
    # Forward pass through the shared encoder (not .encode() — we need gradients)
    anc = F.normalize(model(enc_p["input_ids"].cuda(), enc_p["attention_mask"].cuda()))
    pos = F.normalize(model(enc_h["input_ids"].cuda(), enc_h["attention_mask"].cuda()))

    loss = criterion(anc, pos)
    optimizer.zero_grad()
    loss.backward()
    optimizer.step()

    if step % 500 == 0:
        print(f"step {step:>5d}  loss {loss.item():.4f}")

A single epoch over the ~183K entailment pairs takes roughly 30–60 minutes on a modern GPU. Production sentence transformers train for longer, on larger datasets (AllNLI combines SNLI + MultiNLI for ~1M pairs), and often add a second fine-tuning stage on STS regression data. Even this minimal setup produces embeddings that dramatically outperform raw BERT outputs on similarity tasks.

Step 4: Semantic Search

With a trained model, semantic search is a single matrix operation — encode the corpus once, then compare any query embedding against all stored embeddings.


@torch.no_grad()
def semantic_search(model, tokenizer, corpus, query, top_k=3):
    """Encode a corpus and retrieve the top-k most similar sentences to a query."""
    model.eval()

    # Encode entire corpus in one batch
    enc_c = tokenizer(corpus, padding=True, truncation=True,
                      max_length=128, return_tensors="pt")
    corpus_embs = model.encode(enc_c["input_ids"].cuda(),
                               enc_c["attention_mask"].cuda())  # (N, 768)

    # Encode query
    enc_q = tokenizer([query], padding=True, truncation=True,
                      max_length=128, return_tensors="pt")
    query_emb = model.encode(enc_q["input_ids"].cuda(),
                             enc_q["attention_mask"].cuda())    # (1, 768)

    # Cosine similarities: (1, 768) @ (768, N) -> (1, N)
    scores = (query_emb @ corpus_embs.T).squeeze(0)
    top_indices = scores.argsort(descending=True)[:top_k]

    for idx in top_indices:
        print(f"  [{scores[idx]:.3f}] {corpus[idx]}")

# Example usage
corpus = [
    "A man is eating food.",
    "Someone plays the piano.",
    "A woman is reading a book in the park.",
    "The cat sat on the mat.",
    "A person is playing a musical instrument.",
]
semantic_search(model, tokenizer, corpus, "A person is eating a meal.")
# Expected: "A man is eating food." ranks highest

This project demonstrates the full SBERT pipeline: a pre-trained encoder fine-tuned with a contrastive objective to produce meaningful sentence-level embeddings, and an inference pattern that scales to millions of documents. The key insight is that the architecture is almost unchanged from BERT — the magic is in the training objective. Mean pooling combined with MNR loss transforms a token-level language model into a sentence-level embedding engine.

Production systems extend this recipe with harder negatives (mining or using contradiction pairs from NLI), larger base encoders (RoBERTa-large or DeBERTa-v3), knowledge distillation to smaller models, and matryoshka representation learning that produces embeddings useful at multiple dimensionalities. But the core architecture — shared encoder, mean pooling, contrastive loss — remains exactly what we built here.

When to Choose Encoder-Only Over Generative

The encoder advantage comes from three sources:

Latency. A BERT-Base forward pass on a 128-token input takes ~2ms on a modern GPU. An autoregressive decoder generating a classification answer ("positive") must produce multiple tokens sequentially, each requiring a full forward pass. Even with KV caching, a decoder is 10–100× slower for tasks that produce a single label.

Bidirectionality. Encoders see the full input before making any decision. A sentiment classifier processing "The movie was not good, it was absolutely brilliant" benefits from seeing "absolutely brilliant" while processing "not good." A left-to-right decoder must assign intermediate hidden states before seeing the reversal, making it inherently less efficient at extracting bidirectional features for classification.

Cost. A BERT-Base model with 110M parameters can be deployed on a single CPU in production. A GPT-class model with 7B+ parameters requires a GPU. For high-throughput production workloads processing millions of documents per day — email spam filtering, content moderation, search ranking — the cost difference is orders of magnitude.

The counterargument is flexibility. A large decoder can perform classification zero-shot via prompting, without fine-tuning. This is valuable when labeled data is scarce (fewer than ~100 examples) or when the task definition changes frequently. But when the task is stable and labeled data exists, a fine-tuned encoder is almost always faster, cheaper, and at least as accurate.

Key Insight

Encoder-only models are not obsolete. They occupy a distinct and valuable niche: high-throughput, low-latency discriminative tasks where bidirectional context and fine-tuned precision matter more than generative flexibility. Every major search engine, content moderation system, and enterprise NLP pipeline still relies heavily on encoder-based models in production.

The BERT family established a template that persists today: pre-train a deep Transformer on massive text with a self-supervised objective, then adapt it cheaply to any task. The encoder-only branch of this family tree — BERT, RoBERTa, DeBERTa, and their domain-specific descendants — remains the most practical choice for the majority of real-world NLP deployments. The next chapter traces the decoder-only branch, which sacrificed bidirectionality for the ability to generate text — and ultimately scaled to models that changed how humans interact with computers.
