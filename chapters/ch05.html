Chapter 05

Tokenization and Embeddings

Before a Transformer sees a single word, that word must be broken into tokens and each token must be mapped to a vector. These two operations — tokenization and embedding — are the model's only interface with human language. A poor tokenizer fragments common words into meaningless pieces and burns context window on overhead; a poor embedding table wastes parameters on tokens the model rarely encounters. This chapter traces the evolution from word-level and character-level approaches to the subword methods that power every modern LLM, then explains how the embedding table converts discrete token IDs into the continuous vectors that Transformers actually process.

The Tokenization Problem

Transformers operate on sequences of discrete tokens, not raw text. The choice of how to split text into tokens — the tokenization strategy — has downstream consequences on everything from model quality to inference cost to multilingual fairness. Getting this right is the first and most overlooked design decision.

Why Character-Level Is Too Fine-Grained

Treating each character as a token seems clean — no out-of-vocabulary problem, no arbitrary segmentation decisions. But the cost is prohibitive. The sentence "The Transformer architecture revolutionized NLP" is 47 characters but only 6 words. At character level, the model must process 8× more tokens to represent the same content, consuming quadratic attention memory. Worse, each character carries almost no semantic signal on its own — the model must learn, from scratch, that "T-r-a-n-s-f-o-r-m-e-r" spells a meaningful word. This requires many more layers and much more data to achieve the same quality as a model that starts with word-level or subword-level units.

Character-level models also struggle with long-range dependencies. If the context window is 4,096 tokens, a character-level model sees roughly 4,096 characters — about 2-3 paragraphs of English. A subword model fits 10-15 paragraphs in the same window. The effective information bandwidth per token is dramatically lower.

Why Word-Level Is Too Coarse

Word-level tokenization assigns one token per whitespace-separated word. This works for common English words but fails spectacularly at the edges. Any word not in the fixed vocabulary becomes an unknown token ([UNK]) — and with language's infinite productivity (new names, code identifiers, morphological variants), unknowns are frequent. A 50,000-word vocabulary still produces unknowns for "ChatGPT," "defragmentation," or any non-English word.

Code and mathematical expressions are particularly devastating for word-level tokenizers. A function name like getMaxValue() is a single "word" that no reasonable vocabulary would contain. The expression x**2 + 3*x - 7 mixes operators and identifiers that don't align with natural-language word boundaries. Mathematical notation like $\nabla_\theta \mathcal{L}$ produces entirely out-of-vocabulary symbols. For any model that must handle code, math, or technical content — which includes every modern LLM — word-level tokenization is a non-starter.

Morphologically rich languages amplify the problem further. Finnish or Turkish can generate hundreds of thousands of surface forms from a single root through agglutination, making word-level vocabularies impossibly large. Even German compound nouns (Rindfleischetikettierungsüberwachungsaufgabenübertragungsgesetz) defeat fixed-vocabulary approaches.

The Goldilocks Zone: Subword Tokenization

Subword tokenization finds the sweet spot. Common words like "the" and "transformer" get their own tokens. Rare words are decomposed into frequent subpieces: "revolutionized" might become ["revolution", "ized"]. The vocabulary is finite (typically 32K–200K tokens), but any string can be represented — no unknowns. The key insight is that the right granularity is learned from data: frequently co-occurring character sequences become single tokens, while rare sequences stay decomposed.

Byte Pair Encoding (BPE)

BPE is the most widely used tokenization algorithm in modern language models. Originally a data compression technique (Gage, 1994), Sennrich et al. (2016) adapted it for NLP. The core idea is simple: start with individual characters, count which pairs of characters appear most frequently in the training corpus, merge the most frequent pair into a new symbol, and repeat.

Algorithm: Building the Vocabulary Bottom-Up

BPE builds a vocabulary bottom-up from individual characters. Start with a vocabulary of all characters in the training corpus. Count every adjacent pair of symbols in the corpus. Merge the most frequent pair into a new symbol. Repeat until the vocabulary reaches the desired size. The merge rules — an ordered list of which pairs to combine — are the learned artifact that defines the tokenizer.


# Simplified BPE training
def train_bpe(corpus: list[str], vocab_size: int) -> list[tuple]:
    """Train BPE merge rules from a corpus of words."""
    # Initialize: split each word into characters + end-of-word marker
    word_freqs = {}  # {"l o w ": 5, "l o w e r ": 2, ...}
    for word, freq in count_words(corpus):
        word_freqs[" ".join(word) + " "] = freq

    merges = []
    while len(get_vocab(word_freqs)) < vocab_size:
        # Count all adjacent pairs
        pairs = count_pairs(word_freqs)
        if not pairs:
            break
        # Find and apply the most frequent pair
        best_pair = max(pairs, key=pairs.get)
        word_freqs = merge_pair(best_pair, word_freqs)
        merges.append(best_pair)
    return merges  # Ordered list of merge rules

The merge rules are the learned artifact — they define how to tokenize new text at inference time. To tokenize a new word, start with characters, then apply the merge rules in the order they were learned. This greedy, deterministic process always produces the same tokenization for the same input.

Here is a complete BPE tokenizer that uses the trained merge rules to encode text into token IDs and decode them back.


class BPETokenizer:
    """Minimal BPE tokenizer: trains on corpus, encodes and decodes text."""
    def __init__(self, merges, vocab):
        self.merges = merges                       # list of (str, str) merge pairs
        self.vocab = vocab                         # token -> id mapping
        self.id_to_token = {v: k for k, v in vocab.items()}

    def encode(self, text):
        """Tokenize text by applying learned merge rules in order."""
        tokens = list(text)                        # start with characters
        for pair in self.merges:
            i = 0
            new_tokens = []
            while i < len(tokens):
                # Look for the merge pair at current position
                if i < len(tokens) - 1 and tokens[i] == pair[0] and tokens[i+1] == pair[1]:
                    new_tokens.append(pair[0] + pair[1])  # merge
                    i += 2
                else:
                    new_tokens.append(tokens[i])
                    i += 1
            tokens = new_tokens
        return [self.vocab[t] for t in tokens]     # convert to token ids

    def decode(self, ids):
        """Convert token ids back to text."""
        return "".join(self.id_to_token[i] for i in ids)

# Example usage
tokenizer = BPETokenizer(merges, vocab)
ids = tokenizer.encode("lowest")    # e.g., [low_id, est_id]
text = tokenizer.decode(ids)        # "lowest"

GPT-2 BPE: Byte-Level Foundation

GPT-2 introduced byte-level BPE with a vocabulary of 50,257 tokens. Instead of starting from Unicode characters (which number in the hundreds of thousands), it starts from the 256 byte values. Every possible byte sequence can be represented, so there are never unknown tokens — even binary data or exotic scripts are tokenizable. The byte-level approach also avoids the need for language-specific preprocessing. GPT-2's tokenizer operates directly on UTF-8 encoded bytes.

Tiktoken (GPT-3/4)

OpenAI's tiktoken library implements BPE for GPT-3.5/4 using the cl100k_base vocabulary (100,256 tokens) and GPT-4o uses o200k_base (200,019 tokens). Larger vocabularies improve compression (fewer tokens per text), which means more content fits in the context window. The o200k_base vocabulary is particularly optimized for multilingual text and code, reducing the "token tax" that non-English languages pay with smaller vocabularies.

Fertility: The Multilingual Cost

Fertility is the average number of tokens per word for a given language. English text typically has fertility around 1.3 with GPT-4's tokenizer — most words are one token, some common words need two. But languages with different scripts pay a heavy tax: Chinese might have fertility 2-3× higher (each character becomes multiple tokens), and low-resource languages can see 5-7× the token count for the same semantic content. This means non-English users burn through their context window faster and get slower responses, creating a practical inequality in model utility. Modern tokenizers (o200k_base, LLaMA-3's 128K vocab) reduce this gap by allocating more vocabulary entries to non-English scripts.

WordPiece and SentencePiece

BPE is not the only subword algorithm. Two alternatives — WordPiece and SentencePiece — are used in major models and offer different trade-offs in how they construct vocabularies and handle diverse scripts.

WordPiece

WordPiece (Schuster and Nakajima, 2012) is similar to BPE but selects merges to maximize the likelihood of the training data under a unigram language model, rather than simply picking the most frequent pair. Formally, it merges the pair $(a, b)$ that maximizes $\frac{\text{freq}(ab)}{\text{freq}(a) \cdot \text{freq}(b)}$ — a pointwise mutual information criterion. In practice, the resulting vocabularies are similar to BPE's, but WordPiece tends to prefer merges that reduce the most uncertainty. BERT uses WordPiece with a vocabulary of 30,522 tokens.

The ## Prefix Convention

BERT's tokenizer marks continuation subwords with "##". The word "playing" tokenizes as ["play", "##ing"]. The "##" prefix signals that this piece attaches to the preceding token rather than starting a new word. This convention lets the model distinguish between "##ing" as a suffix and a hypothetical word that starts with "ing." Without this marker, the model would have no way to know whether a token begins a new word or continues the previous one.

SentencePiece

SentencePiece (Kudo and Richardson, 2018) takes a different approach: it operates directly on raw text, treating whitespace as a regular character (represented as "▁"). This is language-agnostic — no pre-tokenization rules are needed for any language. The sentence "I love NLP" becomes ["▁I", "▁love", "▁NL", "P"] — the "▁" encodes the whitespace, allowing perfect reconstruction of the original text including spaces. This design is critical for languages like Chinese and Japanese that don't use whitespace to separate words.

Unigram Language Model Tokenizer

SentencePiece supports both BPE and a unigram language model approach. Instead of building up from characters (bottom-up), the unigram method starts with a large initial vocabulary (often 1M+ candidates) and iteratively removes tokens whose removal least hurts the data likelihood. This top-down approach can produce slightly better compression than BPE for some languages, and it naturally supports multiple tokenizations of the same input — useful for training-time data augmentation. T5 uses the unigram tokenizer within SentencePiece.

Modern Usage and Comparison

LLaMA, LLaMA-2, and Gemma use SentencePiece with BPE. LLaMA-3 switched to tiktoken (byte-level BPE) with a 128K vocabulary. T5 uses SentencePiece with the unigram model. The choice between these is largely engineering preference — the quality differences are small compared to the impact of vocabulary size. What matters more is whether the vocabulary is large enough to cover the target languages efficiently.

Byte-Level Tokenization

The dream of tokenization is universality: a single tokenizer that handles any language, any script, any domain — including code, math, and binary data — without special cases. Byte-level approaches come closest to this ideal.

Byte-Level BPE

GPT-2 pioneered byte-level BPE: instead of starting from Unicode characters, BPE starts from the 256 raw byte values. Any UTF-8 encoded text is a sequence of bytes, so any string — English, Chinese, emoji, code, binary — can be tokenized without special handling. No unknown tokens, ever. The base vocabulary is just 256 entries, and BPE merges build up to the full vocabulary from there.

Compression vs Vocabulary Size Trade-off

A larger vocabulary means better compression (fewer tokens per text) but more embedding parameters (the embedding table is $V \times d_\text{model}$). The relationship is approximately logarithmic: doubling the vocabulary might improve compression by 10-15%, not 100%. At some point, the marginal compression gain from adding another token is smaller than the cost of the extra embedding parameters and the dilution of training signal for rare tokens.

Byte Fallback in SentencePiece

SentencePiece supports a byte fallback mode: if a character cannot be encoded by any learned token, it is decomposed into its UTF-8 byte representation using special byte tokens (<0x41>, <0x42>, etc.). This gives SentencePiece the same "no unknowns" guarantee as byte-level BPE while keeping the primary vocabulary character-based for common scripts. LLaMA and LLaMA-2 use this approach — the vocabulary is mostly character-based for efficiency, but byte tokens serve as a universal fallback for anything unexpected.

Token Embeddings

Tokenization produces a sequence of discrete integer IDs. The Transformer needs continuous vectors. The embedding table bridges this gap — it maps each token ID to a learned $d_\text{model}$-dimensional vector that encodes the token's meaning, usage patterns, and relationships to other tokens.

The Embedding Table

The embedding table is a matrix $E \in \mathbb{R}^{V \times d_\text{model}}$ where $V$ is the vocabulary size and $d_\text{model}$ is the model dimension. Looking up token $i$ retrieves row $i$ of this matrix — a $d_\text{model}$-dimensional vector. This is the token's representation before any Transformer processing. For LLaMA-7B ($V = 32000$, $d_\text{model} = 4096$), the embedding table has 131 million parameters — about 2% of the model's total.

Mathematically, the lookup is equivalent to multiplying a one-hot vector by the embedding matrix: if $\mathbf{e}_i$ is the one-hot vector with a 1 at position $i$, then $\mathbf{e}_i^\top E$ (matrix multiply, $\mathbb{R}^{1 \times V} \times \mathbb{R}^{V \times d_\text{model}} \to \mathbb{R}^{1 \times d_\text{model}}$) retrieves row $i$. In practice, this is implemented as an index operation, not a matrix multiplication — but the gradient flows as if it were a matmul, which is how the embedding rows are learned.

The following PyTorch implementation shows the embedding lookup with $\sqrt{d_\text{model}}$ scaling and optional weight tying to the output projection.


import torch
import torch.nn as nn
import torch.nn.functional as F

class TokenEmbedding(nn.Module):
    """Token embedding with optional weight tying to the output projection."""
    def __init__(self, vocab_size, d_model, tie_weights=True):
        super().__init__()
        self.embedding = nn.Embedding(vocab_size, d_model)  # (vocab_size, d_model)
        self.d_model = d_model
        self.tie_weights = tie_weights
        if not tie_weights:
            self.output_proj = nn.Linear(d_model, vocab_size, bias=False)

    def embed(self, token_ids):
        """Look up embeddings: (batch, seq_len) -> (batch, seq_len, d_model)."""
        return self.embedding(token_ids) * (self.d_model ** 0.5)  # scale by sqrt(d_model)

    def to_logits(self, hidden_states):
        """Project hidden states to vocabulary logits for next-token prediction."""
        if self.tie_weights:
            # Reuse embedding matrix as output projection (weight tying)
            # hidden_states: (batch, seq_len, d_model)
            # embedding.weight: (vocab_size, d_model)
            return hidden_states @ self.embedding.weight.T  # (batch, seq_len, vocab_size)
        return self.output_proj(hidden_states)

# Usage in a language model
emb = TokenEmbedding(vocab_size=32000, d_model=4096, tie_weights=True)
x = emb.embed(token_ids)           # (B, S) -> (B, S, 4096)
# ... transformer layers ...
logits = emb.to_logits(hidden)      # (B, S, 4096) -> (B, S, 32000)
loss = F.cross_entropy(logits.view(-1, 32000), targets.view(-1))

Tied Input/Output Embeddings

Press and Wolf (2017) showed that sharing the embedding matrix between the input embedding layer and the output projection (the linear layer before the final softmax) consistently helps. The output projection maps from $d_\text{model}$ back to vocabulary size — this is exactly $E^\top$ if we tie the weights. Tying saves $V \times d_\text{model}$ parameters and acts as a regularizer by forcing the model to use the same representation space for both input and output. GPT-2, LLaMA, and most modern models tie embeddings.

Embedding Initialization and Scaling

Embeddings are typically initialized from $\mathcal{N}(0, 0.02^2)$ or $\mathcal{N}(0, 1/\sqrt{d_\text{model}})$. The original Transformer multiplied embeddings by $\sqrt{d_\text{model}}$ before adding positional encodings, scaling them up so they are not dominated by the positional signal. This scaling is essential: without it, the positional encoding (which has unit variance by construction) would overwhelm the embedding signal in early training.

The Embedding Dimension Choice

Common values of $d_\text{model}$ follow a power-of-two progression: 512 (original Transformer), 768 (BERT-base, GPT-2 small), 1024 (BERT-large), 2048, 4096 (LLaMA-7B), 8192 (LLaMA-70B). Larger dimensions give each token more expressive capacity but cost more in parameters and compute. Scaling laws suggest that model width and depth should grow together — a very wide but shallow model or a narrow but deep model are both suboptimal for a fixed parameter budget.

Token Frequency and Embedding Quality

Tokens that appear frequently in training data develop rich, well-separated embeddings because they receive many gradient updates. Rare tokens — unusual names, code snippets, non-Latin characters — receive few updates, and their embeddings remain close to their random initialization. This creates a quality gap: the model is far more capable at processing common tokens than rare ones.

Techniques like vocabulary pruning (removing unused tokens) and adaptive input representations (giving common tokens more dimensions) can partially address this, but the fundamental tension between vocabulary coverage and per-token training signal remains.

Special Tokens

Beyond content tokens, every model reserves a handful of IDs for control signals that structure the input, mark boundaries, and coordinate different phases of processing. These special tokens are critical infrastructure — get them wrong and the model misinterprets its input.

Encoder-Style Special Tokens

BERT introduced three special tokens that became standard for encoder models. [CLS] (classification) is prepended to every input and serves as a sequence-level representation after encoding — classification heads read from [CLS]'s final hidden state. [SEP] (separator) marks the boundary between sentence pairs in tasks like natural language inference. [MASK] is used during pre-training to mark positions where the model must predict the original token.

Decoder-Style Special Tokens

Autoregressive models use different conventions. <BOS> (beginning of sequence) marks the start of generation — the model's first prediction is conditioned on this token alone. <EOS> (end of sequence) signals that generation should stop. <PAD> fills sequences to a uniform length within a batch; padding tokens are masked out in the attention computation so they don't influence predictions.

Modern Special Tokens

As models have become more capable, the special token vocabulary has grown. Chat models add tokens for turn boundaries: <|im_start|> and <|im_end|> delimit messages in ChatGPT's format. System prompt tokens mark instructions that should be treated differently from user content. Tool call delimiters bracket structured outputs like function calls. Multimodal models add <image> placeholder tokens where visual encodings will be inserted. Each special token must be added to the vocabulary, assigned an embedding, and given appropriate handling in the attention mask.

Key Insight

Tokenization is the most under-appreciated design decision in modern LLMs. A poor tokenizer can cost a model 30-50% of its effective context window for non-English languages, fragment code in semantically meaningless ways, and create artifacts where common phrases split across token boundaries receive inconsistent treatment. The shift toward larger, more multilingual vocabularies (100K-200K tokens) is one of the quietest but most impactful improvements in recent model generations.

This end-to-end pipeline ties everything together: tokenize raw text, prepend and append special tokens, pad to uniform length, build the attention mask, and produce embeddings ready for the Transformer.


import torch
import torch.nn as nn

class TextPipeline:
    """End-to-end: raw text -> tokenized + padded batch -> embeddings."""
    def __init__(self, tokenizer, d_model, max_len=512,
                 pad_id=0, cls_id=101, sep_id=102):
        self.tokenizer = tokenizer
        self.max_len = max_len
        self.pad_id = pad_id
        self.cls_id = cls_id
        self.sep_id = sep_id
        self.embedding = nn.Embedding(tokenizer.vocab_size, d_model)
        self.pos_embedding = nn.Embedding(max_len, d_model)

    def tokenize_batch(self, texts):
        """Tokenize a list of strings into padded tensors with attention masks."""
        batch_ids = []
        for text in texts:
            ids = [self.cls_id] + self.tokenizer.encode(text) + [self.sep_id]
            ids = ids[:self.max_len]                          # truncate
            batch_ids.append(ids)

        # Pad to longest sequence in batch
        max_len = max(len(ids) for ids in batch_ids)
        padded = [ids + [self.pad_id] * (max_len - len(ids)) for ids in batch_ids]

        input_ids = torch.tensor(padded)                      # (batch, seq_len)
        attention_mask = (input_ids != self.pad_id).long()     # (batch, seq_len)
        return input_ids, attention_mask

    def embed(self, input_ids):
        """Convert token ids to embeddings with positional encoding."""
        positions = torch.arange(input_ids.size(1), device=input_ids.device)
        tok_emb = self.embedding(input_ids)                   # (batch, seq_len, d_model)
        pos_emb = self.pos_embedding(positions)               # (seq_len, d_model)
        return tok_emb + pos_emb                              # (batch, seq_len, d_model)

# Full pipeline
input_ids, mask = pipeline.tokenize_batch(["The cat sat.", "Hello world."])
embeddings = pipeline.embed(input_ids)  # ready for transformer layers
