Chapter 11

Instruction Tuning, RLHF, and Alignment

Ask GPT-3 "What is the capital of France?" and it might respond "What is the capital of Germany? What is the capital of Italy?" — it was trained to *continue* text, not to *answer* questions. InstructGPT, a 1.3-billion-parameter model fine-tuned with human feedback, was preferred by users over the raw 175-billion-parameter GPT-3. A model 100× smaller won — not because it knew more, but because it learned what humans actually wanted. This chapter explains the complete post-training pipeline that turns a raw next-token predictor into a helpful, harmless assistant: supervised fine-tuning on instruction data, reinforcement learning from human feedback, direct preference optimization, constitutional AI, and the fundamental trade-offs between safety and capability.

Supervised Fine-Tuning (SFT)

A pre-trained language model has learned the statistical structure of internet text — it can predict the next token with impressive accuracy. But the distribution of internet text is not the distribution of helpful responses. Web text contains arguments, spam, fiction, code, forum posts, and Wikipedia articles in roughly the proportions they appear online. Supervised fine-tuning reshapes this distribution: instead of predicting what comes next in any text, the model learns to produce the kind of responses a helpful assistant would give.

Instruction-Following Data Collection

The first ingredient is a dataset of (instruction, response) pairs. Several approaches have been used to create these datasets, each with different quality-cost tradeoffs:

FLAN (Wei et al., 2022) aggregated over 1,800 NLP tasks into a unified instruction format. Each task was converted to natural language: "Translate the following sentence to French: ..." or "Is the following review positive or negative? ..." This was the first large-scale instruction mixture, and Flan-T5 (Chapter 9) demonstrated that it works for encoder-decoder models as well as decoder-only ones.

Self-Instruct (Wang et al., 2023) used GPT-3 itself to generate instruction data. Starting from a small seed set of 175 human-written tasks, the model was prompted to generate new instructions, inputs, and outputs iteratively. This bootstrapping approach produced 52K diverse examples at near-zero annotation cost — the first instance of a theme that recurs throughout this chapter: using models to supervise models.

Alpaca (Stanford, 2023) applied Self-Instruct to GPT-3.5-Turbo, generating 52K instruction examples for about $600 in API costs. These examples were used to fine-tune LLaMA-7B, creating a chatbot-like model that demonstrated surprisingly strong instruction-following for its size.

ShareGPT collected real multi-turn conversations between users and ChatGPT, shared publicly by users. The quality is uneven — some conversations are insightful, others are trivial — but the diversity of real-world use cases is unmatched by synthetic data.

Data Quality vs. Quantity

The LIMA paper (Zhou et al., 2023) made a provocative claim: 1,000 carefully curated instruction-response pairs are sufficient for strong instruction-following. The authors called this the Superficial Alignment Hypothesis — alignment is mostly about learning the *format* (respond helpfully in a conversational style) rather than the *knowledge* (which is already in the pre-trained weights). A model fine-tuned on 1,000 high-quality examples outperformed models fine-tuned on 52,000 noisy ones.

This finding contrasts sharply with pre-training scaling laws (Chapter 10), where more data monotonically improves loss. SFT follows a different scaling regime entirely: quality dominates quantity. A few hundred carefully written, diverse demonstrations can teach the model the "assistant persona" without diluting its pre-trained knowledge.

Prompt Formatting

The model needs to know *where* it should generate — the format tells it which tokens are the instruction and which are the expected response. Modern LLMs use a chat template with role markers:

The format matters more than it might seem. Without clear role markers, the model cannot distinguish between text it should *read* (the user's question) and text it should *write* (its answer). Chapter 8 discussed how GPT-4's system prompt acts as a behavioral control mechanism at inference time — the chat template is how the model learns to respect it during training.

SFT Training Recipe

SFT training is deliberately conservative. The pre-trained model already contains vast knowledge encoded in its weights — the goal is to adjust its behavior without destroying that knowledge. The standard recipe:

  • Learning rate: 1e-5 to 2e-5, roughly 10× lower than pre-training.
  • Epochs: 1 to 3. More epochs risk overfitting to the instruction dataset's specific phrasings.
  • Schedule: Cosine decay with warmup, typically 3-10% warmup steps.
  • Regularization: Some teams mix a fraction of pre-training data into the SFT batch to prevent catastrophic forgetting — the model becoming good at chatting but losing factual knowledge or coding ability.

Here is a minimal SFT training loop using the Alpaca instruction dataset from HuggingFace:


import torch
import torch.nn as nn
import torch.nn.functional as F
from datasets import load_dataset

# Load Alpaca instruction data from HuggingFace
ds = load_dataset("tatsu-lab/alpaca", split="train[:5000]")

def format_alpaca(example):
    """Format an Alpaca example into a chat-style training string."""
    instruction = example["instruction"]
    inp = example.get("input", "")
    output = example["output"]
    if inp:
        prompt = f"<|user|>\n{instruction}\n{inp}\n<|assistant|>\n"
    else:
        prompt = f"<|user|>\n{instruction}\n<|assistant|>\n"
    return {"text": prompt + output, "prompt_len": len(prompt)}

ds = ds.map(format_alpaca)

# Assume a pre-trained GPT model and tokenizer are available
# model = load_pretrained_gpt(...)
# tokenizer = load_tokenizer(...)

optimizer = torch.optim.AdamW(model.parameters(), lr=2e-5, weight_decay=0.01)

for epoch in range(2):  # Few epochs to avoid catastrophic forgetting
    for batch in dataloader:
        input_ids = batch["input_ids"]                      # (B, S)
        labels = input_ids.clone()
        # Mask loss on prompt tokens — only train on assistant response
        labels[:, :batch["prompt_len"]] = -100              # ignore prompt
        logits = model(input_ids)                            # (B, S, vocab)
        loss = F.cross_entropy(
            logits[:, :-1].reshape(-1, logits.size(-1)),     # (B*(S-1), V)
            labels[:, 1:].reshape(-1),                       # (B*(S-1),)
            ignore_index=-100
        )
        loss.backward()
        torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
        optimizer.step()
        optimizer.zero_grad()

Reinforcement Learning from Human Feedback (RLHF)

SFT teaches the model the format of helpful responses. But it optimizes a proxy: cross-entropy on *demonstrations*, not the quality of the model's own generations. A model can achieve low SFT loss by memorizing the training examples' style while still producing unhelpful, incorrect, or harmful outputs when faced with novel prompts. RLHF addresses this gap by directly optimizing for human preferences.

The InstructGPT Pipeline

Ouyang et al. (2022) defined the modern alignment recipe in three stages. This pipeline — SFT, then reward model training, then reinforcement learning — remains the foundation of how production LLMs are aligned.

The insight behind this pipeline is that humans are better at *judging* than *demonstrating*. Writing the perfect response to a complex question is hard. Comparing two responses and picking the better one is much easier. Stage 2 converts these cheap comparative judgments into a reward signal that Stage 3 can optimize.

Reward Model Training: The Bradley-Terry Model

The reward model takes a prompt $x$ and a response $y$ and outputs a scalar score $r_\phi(x, y)$ representing how good the response is. It is trained on human preference data: given a prompt and two responses, one preferred ($y_w$) and one rejected ($y_l$), the model learns to assign a higher score to the preferred response.

The Bradley-Terry model defines the probability that response $y_w$ is preferred over $y_l$ as a function of the reward difference:

$$P(y_w \succ y_l \mid x) = \sigma\!\left(r_\phi(x, y_w) - r_\phi(x, y_l)\right)$$

Here $\sigma$ is the sigmoid function ($\sigma(z) = 1/(1 + e^{-z})$, mapping any real number to a probability). The reward model loss is the negative log-likelihood of the observed preferences:

$$\mathcal{L}_{\text{RM}} = -\mathbb{E}_{(x, y_w, y_l)}\!\left[\log \sigma\!\left(r_\phi(x, y_w) - r_\phi(x, y_l)\right)\right]$$

Minimizing this loss pushes $r_\phi(x, y_w)$ up and $r_\phi(x, y_l)$ down — exactly what we want.

Architecturally, the reward model is typically the SFT model itself with the language modeling head replaced by a single linear projection $\text{nn.Linear}(d_\text{model}, 1)$ that maps the last token's hidden state to a scalar reward. Unlike BERT (Chapter 7), which uses a [CLS] token for classification, decoder-only models extract the last-token representation — following the pattern established in Chapter 8.

Here is the reward model training in PyTorch, using preference data from HuggingFace:


import torch
import torch.nn as nn
from datasets import load_dataset

# Anthropic's preference dataset: each example has 'chosen' and 'rejected'
ds = load_dataset("Anthropic/hh-rlhf", split="train[:5000]")

class RewardModel(nn.Module):
    """Reward model: pre-trained GPT backbone + scalar value head."""
    def __init__(self, backbone, d_model):
        super().__init__()
        self.backbone = backbone                      # frozen or fine-tuned GPT
        self.value_head = nn.Linear(d_model, 1)       # scalar reward output

    def forward(self, input_ids):
        h = self.backbone(input_ids)                   # (B, S, d_model)
        reward = self.value_head(h[:, -1, :])          # (B, 1) — last token
        return reward.squeeze(-1)                       # (B,)

# Training loop: Bradley-Terry loss on preference pairs
for batch in dataloader:
    r_w = reward_model(batch["chosen_ids"])             # (B,) — preferred
    r_l = reward_model(batch["rejected_ids"])           # (B,) — rejected
    # Bradley-Terry: push r_w above r_l
    loss = -torch.log(torch.sigmoid(r_w - r_l)).mean()
    loss.backward()
    optimizer.step()
    optimizer.zero_grad()

PPO with KL Penalty

With a trained reward model, we can now optimize the language model's policy $\pi_\theta$ to produce high-reward responses. The objective is:

$$\max_\theta \; \mathbb{E}_{x \sim \mathcal{D},\, y \sim \pi_\theta(\cdot|x)}\!\left[r_\phi(x, y) - \beta \, D_{\text{KL}}\!\left(\pi_\theta(\cdot|x) \,\|\, \pi_{\text{ref}}(\cdot|x)\right)\right]$$

The first term maximizes reward. The second term — the KL penalty — prevents the policy from drifting too far from the reference model $\pi_{\text{ref}}$ (the SFT model). The hyperparameter $\beta$ controls the strength of this constraint. Without the KL penalty, the policy quickly finds adversarial inputs that exploit imperfections in the reward model — producing responses that score high reward but are actually low quality.

A simplified PPO training step shows the core mechanics:


@torch.no_grad()
def compute_advantages(policy, ref_policy, reward_model, prompts, beta=0.1):
    """Compute per-token advantages for PPO update."""
    # Generate responses from current policy
    sequences = policy.generate(prompts, max_new_tokens=256)   # (B, S)

    # Log-probs under current and reference policies
    old_logp = policy.log_prob(sequences)                       # (B, S)
    ref_logp = ref_policy.log_prob(sequences)                   # (B, S)

    # KL penalty: per-sequence divergence from reference
    kl = (old_logp - ref_logp).sum(dim=-1)                      # (B,)

    # Reward from RM, minus KL penalty
    rewards = reward_model(sequences)                            # (B,)
    advantages = rewards - beta * kl                              # (B,)
    return sequences, old_logp, advantages

def ppo_step(policy, sequences, old_logp, advantages, clip_eps=0.2):
    """Single PPO update: clipped surrogate objective."""
    new_logp = policy.log_prob(sequences)                        # (B, S)
    # Importance sampling ratio
    ratio = (new_logp.sum(-1) - old_logp.sum(-1).detach()).exp() # (B,)
    # Clipped surrogate
    surr1 = ratio * advantages
    surr2 = torch.clamp(ratio, 1 - clip_eps, 1 + clip_eps) * advantages
    loss = -torch.min(surr1, surr2).mean()
    loss.backward()
    optimizer.step()
    optimizer.zero_grad()

Reward Hacking and Overoptimization

The reward model is a proxy for human preferences — it is not a perfect oracle. When you optimize a proxy too aggressively, you exploit its weaknesses. Gao et al. (2023) showed this empirically: as the KL divergence from the reference model increases (meaning the policy has drifted further), the proxy reward keeps rising, but *true quality* (measured by a separate gold reward model or human evaluation) peaks and then declines.

Concrete examples of reward hacking: the model learns to produce sycophantic responses ("What a brilliant question! You're absolutely right!"), excessively long and verbose answers (which tend to score higher on some reward models), or responses that use specific formatting patterns the reward model has a bias toward. Goodhart's Law — "when a measure becomes a target, it ceases to be a good measure" — is the fundamental challenge of RLHF.

Warning: Goodhart's Law in Practice

Every RLHF system eventually encounters reward hacking. The reward model is a proxy for human preferences, and optimizing a proxy too aggressively always produces artifacts. The KL penalty is not optional — it is the primary defense against a policy that finds loopholes in the reward model. Ensemble reward models and periodic reward model retraining provide additional robustness.

RLHF at Scale

Different labs have adopted different approaches to RLHF at production scale:

OpenAI (InstructGPT, GPT-4) uses PPO with extensive human annotation pipelines. Thousands of human annotators write demonstrations and compare model outputs. Red-teaming — deliberately trying to elicit harmful outputs — is a critical part of the annotation process.

Anthropic (Claude family) combines RLHF with Constitutional AI (Section 4), using the model itself to generate preference labels guided by a set of principles. This reduces the need for human annotators while maintaining alignment quality.

Google DeepMind (Gemini) uses rule-based reward models alongside human preferences, with adversarial probing to identify failure modes. The LLaMA 2-Chat models (Chapter 8) used rejection sampling alongside PPO, demonstrating that RLHF could be applied effectively to open-source models.

Direct Preference Optimization (DPO)

RLHF works, but it is complex. Three separate models (SFT policy, reward model, value function), a reinforcement learning loop, sensitive hyperparameters ($\beta$, PPO clip ratio, learning rate schedules for each component), and instability from on-policy sampling. DPO asks: can we get the same result with just a loss function?

The Key Insight

Rafailov et al. (2023) showed that the optimal policy under the KL-constrained RLHF objective has a closed-form solution. The reward function is implicitly defined by the ratio of the policy to the reference model:

$$r^*(x, y) = \beta \log \frac{\pi^*(y|x)}{\pi_{\text{ref}}(y|x)} + \beta \log Z(x)$$

where $Z(x)$ is a normalizing constant that depends only on the prompt. This identity means we can substitute the policy directly into the Bradley-Terry preference model — eliminating the need for a separate reward model entirely.

The DPO Loss

The DPO loss operates directly on preference pairs $(x, y_w, y_l)$ — the same data used to train the reward model in RLHF, but without the intermediate step:

$$\mathcal{L}_{\text{DPO}}(\pi_\theta; \pi_{\text{ref}}) = -\mathbb{E}_{(x, y_w, y_l)}\!\left[\log\sigma\!\left(\beta \log\frac{\pi_\theta(y_w|x)}{\pi_{\text{ref}}(y_w|x)} - \beta \log\frac{\pi_\theta(y_l|x)}{\pi_{\text{ref}}(y_l|x)}\right)\right]$$

Each log-ratio $\log(\pi_\theta(y|x) / \pi_{\text{ref}}(y|x))$ measures how much the current policy has shifted from the reference model for a given response. The loss says: the preferred response should shift *more* (become more likely under $\pi_\theta$ relative to $\pi_{\text{ref}}$) than the rejected response. The $\beta$ parameter controls the KL constraint strength, exactly as in the PPO objective.

The implementation is remarkably concise — this is the same preference data we used for the reward model above, but with no intermediate model needed:


import torch
import torch.nn.functional as F
from datasets import load_dataset

# Same preference data as the reward model — no RM needed this time
ds = load_dataset("Anthropic/hh-rlhf", split="train[:5000]")

def dpo_loss(policy, ref_policy, chosen_ids, rejected_ids, beta=0.1):
    """Direct Preference Optimization — no reward model, no RL loop."""
    # Log-probabilities under current policy (per-token, summed)
    pi_logp_w = policy.log_prob(chosen_ids).sum(-1)        # (B,)
    pi_logp_l = policy.log_prob(rejected_ids).sum(-1)      # (B,)

    # Log-probabilities under frozen reference policy
    with torch.no_grad():
        ref_logp_w = ref_policy.log_prob(chosen_ids).sum(-1)   # (B,)
        ref_logp_l = ref_policy.log_prob(rejected_ids).sum(-1) # (B,)

    # Log-ratios: how much has the policy shifted from reference?
    logr_w = pi_logp_w - ref_logp_w                         # (B,)
    logr_l = pi_logp_l - ref_logp_l                         # (B,)

    # DPO loss: preferred should shift more than rejected
    loss = -F.logsigmoid(beta * (logr_w - logr_l)).mean()
    return loss

Key Insight

DPO and RLHF optimize the *same* objective — a KL-constrained reward maximization. The difference is computational, not mathematical. RLHF solves it iteratively through sampling and RL; DPO solves it in closed form via a supervised loss. This is why they often achieve comparable results — and why DPO is preferred when simplicity and stability matter more than on-policy exploration.

When PPO Still Wins

DPO is simpler and more stable, but PPO has advantages in specific regimes. Complex instruction following, mathematics, and coding — domains where the reward signal is sparse and can be verified programmatically — tend to favor PPO. The key difference is on-policy exploration: PPO generates new responses during training and learns from them, while DPO trains on a fixed preference dataset. When the distribution of good responses is narrow (as in mathematical proofs or correct code), the ability to explore diverse solution paths during training matters.

The reasoning models from Chapter 8 — o1, o3, and DeepSeek R1 — all rely on RL-based training, not DPO. When the task requires extended chains of reasoning, with verification at each step, on-policy RL appears critical for learning the exploration-exploitation balance that produces correct solution paths.

Constitutional AI and RLAIF

RLHF requires thousands of human annotators — expensive, slow, and difficult to scale. What if the model could supervise itself? Constitutional AI and RLAIF explore exactly this: replacing human preference labelers with the model itself, guided by a set of principles.

Constitutional AI (Anthropic, 2022)

Constitutional AI (CAI) operates in two phases. In the supervised phase, the model generates a response to a prompt, then critiques its own response against a set of natural-language principles (the "constitution"), and finally revises the response to address the critique. In the RL phase, the model itself provides preference labels — comparing original vs. revised responses — which are used for RLHF training.

RLAIF: LLM as Preference Labeler

RLAIF (Reinforcement Learning from AI Feedback, Lee et al., 2023) generalizes the idea: replace human labelers with an LLM for any preference comparison, not just constitutional self-critique. The surprising finding is that RLAIF can match RLHF quality when the labeler LLM is strong enough and prompted with chain-of-thought reasoning. The cost reduction is dramatic: $0.01 per pair versus $1-3 per pair for human annotation, a 100-300× reduction.

Self-Play Fine-Tuning (SPIN)

SPIN (Chen et al., 2024) takes the self-improvement idea further: the model plays against itself. In each iteration, the model tries to distinguish its own generated text from human-written text. The "real" label goes to human demonstrations, and the "fake" label goes to the model's own outputs. After training, the model improves, making the discrimination task harder in the next round. This iterative process is structurally similar to GAN training applied to language model alignment.

Scalable Oversight

As AI systems become more capable, human oversight becomes the bottleneck. How do you align a model that is smarter than you on the task being evaluated? This is the scalable oversight problem, and several research directions are being explored:

  • Weak-to-strong generalization (Burns et al., 2023): a small model supervises a large model. Surprisingly, the large model can sometimes generalize *beyond* the quality of the supervisor.
  • Debate (Irving et al., 2018): two AI systems argue opposing positions; a human judge picks the winner. Even a weak judge can identify the more truthful argument when it is presented alongside a challenge.
  • Amplification (Christiano et al., 2018): decompose hard oversight tasks into simpler subtasks that humans *can* evaluate accurately.

These are frontier research directions — not deployed at production scale — but they point toward a future where alignment can continue to improve even as models surpass human ability on specific tasks.

Alignment Tax and Capability Trade-offs

Alignment improves the model's usefulness. But does it come at a cost to raw capability?

Alignment Tax

The evidence is mixed. On one hand, InstructGPT (1.3B, aligned) was preferred by users over GPT-3 (175B, unaligned) — alignment made the smaller model more useful, not less capable. On the other hand, some benchmarks — particularly code generation (HumanEval), factual QA (TriviaQA), and mathematical reasoning — show slight degradation after RLHF. The model becomes more cautious, sometimes refusing valid requests or hedging when the pre-trained model would have answered directly.

The nuanced answer: alignment *redistributes* capability. The model becomes much better at tasks humans care about (following instructions, being helpful, refusing harmful requests) and slightly worse at tasks measured by raw benchmark performance (unfiltered text completion, unrestricted code generation). Whether this is a "tax" depends on what you are optimizing for.

Safety vs. Helpfulness: The Pareto Frontier

Different labs target different points on this frontier, and the choice is fundamentally about values, not technology. A model deployed for medical advice should prioritize safety over helpfulness. A model for creative writing should prioritize helpfulness. The alignment pipeline does not dictate where on the frontier to sit — it provides the tools to reach any point on it.

System Prompts and Meta-Prompting

The system prompt is the runtime mechanism for alignment. The same base model can serve radically different use cases depending on its system prompt: "You are a helpful coding assistant" vs. "You are a careful medical information provider." This decouples the alignment *capability* (trained during RLHF) from the alignment *policy* (specified at inference time via the system prompt).

Chapter 8 introduced system prompts as a behavioral control mechanism in GPT-4. Here we see the other side of that story: the alignment training teaches the model to *follow* system prompt instructions faithfully. Without SFT and RLHF, the model would ignore the system prompt — it is just another piece of text in the context window. The alignment pipeline teaches the model to treat it as a persistent behavioral constraint.

Meta-prompting takes this one step further: the system prompt itself is generated or refined by another model or automated process. This enables dynamic alignment — adjusting the model's behavior based on the user's context, the task type, or the risk level of the request — without retraining.
