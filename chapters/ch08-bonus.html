Chapter 8 Bonus

Project: Building a Mixture of Experts Language Model from Scratch

Mixtral 8×7B has 47 billion parameters but its forward pass costs the same as a 13B dense model. DeepSeek V3 is a 671B model that activates 37B per token. That arithmetic — total parameters decoupled from active parameters — is the entire promise of sparse Mixture of Experts. The papers describe it cleanly. The actual training is a different story: routers collapse onto two experts within a hundred steps, gradients refuse to flow through the top-k operator, and the load-balance auxiliary loss has to be tuned to a tenth of a percent or the model never learns anything. By the end of this chapter you will have built a working MoE language model in pure PyTorch, trained it on TinyStories on a single A100 in under an hour, and watched expert specialization emerge in real time — one expert quietly claiming punctuation, another firing on dialogue, a third on character names — with no supervision telling them what to specialize on.

The rest of the chapter stacks ideas in order: what MoE buys you, then each trainable piece, then how training and inference close the loop. The figure below is a single reference map — use it to see where the current section sits, and come back when a detail needs a home in the whole system. Read the left column depth stack as L Transformer blocks (attention + FFN + residuals, as in Chapter 8); the small D / M labels refer only to the FFN half of each block, not to attention.

The Conditional-Computation Contract

A dense transformer pays for every parameter on every token. Add an FFN of width $d_{ff}$ to a model and every single token in every batch flows through that FFN's $2\, d_{model}\, d_{ff}$ weights (matrix multiply, $\mathbb{R}^{n \times d_{model}} \times \mathbb{R}^{d_{model} \times d_{ff}} \to \mathbb{R}^{n \times d_{ff}}$). Capacity and compute scale together. To get a bigger model you must accept proportionally bigger FLOPs.

Mixture of Experts breaks that tie. Replace the single FFN with $N$ identically-shaped expert FFNs and a small router network. The router reads each token and picks the top $k$ experts for it — usually $k = 2$ out of $N = 8$. The token only flows through those $k$ experts. The other $N - k$ experts sit idle for that token. Total parameters scale with $N$. Active parameters per token scale with $k$. The reader from Chapter 8 already saw the consequences: Mixtral fits its weights in 8 expert FFNs but only uses 2 per token, so a 47B model runs as fast as a 13B one.

The active-parameter accounting follows directly. For an MoE layer with $N$ experts, top-$k$ routing, and $n_s$ shared experts (always-on, used by Mixtral's successors and DeepSeek), each token sees:

$$\mathrm{active\,FFN\,params\,per\,token} = (k + n_s) \cdot \bigl(2\, d_{model}\, d_{ff} + d_{model}\, d_{ff}\bigr)$$

The total parameters live in $N$ routed experts plus $n_s$ shared experts plus a tiny router matrix of shape $\mathbb{R}^{d_{model} \times N}$. The ratio $(k + n_s) / (N + n_s)$ is the activation fraction: Mixtral 8×7B sits at $2 / 8 = 25\%$, DeepSeek V3 at $9 / 257 \approx 3.5\%$. Smaller activation fractions mean more parameters per FLOP — more knowledge stored per unit of inference cost.

What follows is an honest implementation of that machinery. We will build each component — expert, router, dispatch, load-balance loss, full block, training loop, KV-cache inference, evaluation — in a few hundred lines of pure PyTorch. The point is not a production-grade kernel; it is a model you can step through with a debugger and modify in a single file.

The Expert: One SwiGLU FFN

An expert in a modern MoE is just a feed-forward network. There is no architectural difference between an expert and the FFN inside a dense LLaMA block — same SwiGLU activation, same linear layers, same shape. What differs is that there are $N$ of them and a router that picks which ones run.

SwiGLU (Shazeer, 2020) is the activation that LLaMA, Mistral, Mixtral, and DeepSeek all use. It is a gated variant of Swish. Given input $x \in \mathbb{R}^{d_{model}}$ and three weight matrices $W_1, W_3 \in \mathbb{R}^{d_{model} \times d_{ff}}$ and $W_2 \in \mathbb{R}^{d_{ff} \times d_{model}}$:

$$E(x) = (\mathrm{SiLU}(x W_1) \odot x W_3)\, W_2$$

The juxtaposition $x W_1$ is matrix multiply (in the per-token view, vector $\times$ matrix, $\mathbb{R}^{d_{model}} \times \mathbb{R}^{d_{model} \times d_{ff}} \to \mathbb{R}^{d_{ff}}$). The $\odot$ is element-wise multiplication (Hadamard product, both operands $\in \mathbb{R}^{d_{ff}}$): $W_3$ produces a gate that the SiLU branch is multiplied against, dimension-by-dimension. The final $W_2$ projects back to $d_{model}$. The cost is three projections instead of two — SwiGLU spends 1.5× the FLOPs of a vanilla ReLU FFN at the same $d_{ff}$, which is why papers often quote SwiGLU FFNs at $d_{ff} = \tfrac{2}{3} \cdot 4\, d_{model}$ to keep the parameter count comparable.


import torch
import torch.nn as nn
import torch.nn.functional as F

class SwiGLUExpert(nn.Module):
    """A single expert: a SwiGLU feed-forward network. Identical in shape
    to a LLaMA / Mixtral FFN — what makes it an 'expert' is that there are
    N of them and a router decides which ones run for each token."""
    def __init__(self, d_model: int, d_ff: int):
        super().__init__()
        self.w1 = nn.Linear(d_model, d_ff, bias=False)  # gate-up branch
        self.w3 = nn.Linear(d_model, d_ff, bias=False)  # value-up branch
        self.w2 = nn.Linear(d_ff, d_model, bias=False)  # down

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        # x: (..., d_model) — works on (B, T, d) or flattened (BT, d)
        gate = F.silu(self.w1(x))                       # (..., d_ff)
        value = self.w3(x)                              # (..., d_ff)
        return self.w2(gate * value)                    # (..., d_model)

One detail worth noting: the expert takes inputs of arbitrary leading shape. We will exploit this in Section 4 by flattening a $(B, T, d_{model})$ batch into $(BT, d_{model})$ and feeding each expert only the rows of tokens that the router assigned to it. The expert does not care — its forward pass is just three matrix multiplies and an element-wise gate.

The Router: Top-k Gating

The router is the smallest module in the model and the hardest to train. It is a single linear layer of shape $\mathbb{R}^{d_{model} \times N}$ that produces $N$ logits per token — one score per expert. We then pick the top $k$ scores, renormalize them with a softmax, and use those $k$ weights to combine the chosen experts' outputs.

$$g(x) = \mathrm{softmax}_k\bigl(\mathrm{top\text{-}}k(x W_g)\bigr) \in \mathbb{R}^{k}$$

Three subtleties worth pausing on. First, the softmax is over the selected $k$ logits, not all $N$. This is the Switch Transformer convention: the chosen experts' weights sum to one regardless of how confidently they were chosen. Second, the top-$k$ operation is non-differentiable — it is a hard selection, not a soft attention. Gradients flow into the gate only through the $k$ weights that were actually used; the other $N - k$ logits get no gradient signal from this token. Third, the gate weight $g_i(x)$ is a scalar that scales the corresponding expert output $E_i(x)$: $g_i(x)\, E_i(x)$ is scalar times vector, the standard scalar multiplication.


class TopKRouter(nn.Module):
    """Top-k softmax router. Returns the chosen expert indices, the
    renormalized routing weights, and the full routing probabilities
    (which the load-balance loss needs)."""
    def __init__(self, d_model: int, n_experts: int, k: int):
        super().__init__()
        self.n_experts = n_experts
        self.k = k
        self.gate = nn.Linear(d_model, n_experts, bias=False)

    def forward(self, x_flat: torch.Tensor):
        # x_flat: (BT, d_model) — a flat batch of tokens
        logits = self.gate(x_flat)                          # (BT, n_experts)

        # Full softmax over ALL experts — needed for the load-balance loss
        full_probs = F.softmax(logits, dim=-1)              # (BT, n_experts)

        # Top-k selection then renormalize over the selected k
        topk_logits, topk_idx = logits.topk(self.k, dim=-1) # both (BT, k)
        topk_weights = F.softmax(topk_logits, dim=-1)       # (BT, k), rows sum to 1

        return topk_idx, topk_weights, full_probs, logits

We return four tensors. The pair (topk_idx, topk_weights) is what dispatch needs. The full_probs tensor is what the load-balance loss needs — it has to see the soft routing distribution over all $N$ experts, not just the chosen $k$, to push the average mass toward uniform. The raw logits are what the router z-loss needs to penalize. We will get to both losses in Section 5.

Combining Experts: The Scatter-Gather Dispatch

The router has produced, for each of $BT$ tokens, the indices of $k$ experts and a weight per expert. Now we have to actually run those experts and assemble the output. There are roughly three ways to do this. The fast way uses a fused kernel like Megablocks that turns the dispatch into one big block-sparse matrix multiply. The naive way runs each expert on the full batch and zeros out the irrelevant rows — trivial to write but throws away the FLOPs win. The middle way, which we will use, loops over experts and feeds each one only the rows of tokens that selected it.

The pattern is straightforward. Flatten the $(B, T, d_{model})$ activations into $(BT, d_{model})$. For each expert $i$, find the rows where $i$ appears in the top-$k$ indices, gather just those rows, run the expert, multiply by the corresponding routing weights, and scatter-add the results back into a zero buffer. Each token contributes $k$ additions to the output buffer — one per expert it routed to.


class MoELayer(nn.Module):
    """Routes each token to its top-k experts and combines outputs.
    Pedagogically clear loop-over-experts dispatch; not a fast kernel."""
    def __init__(self, d_model: int, d_ff: int, n_experts: int, k: int,
                 n_shared: int = 0):
        super().__init__()
        self.n_experts = n_experts
        self.k = k
        self.router = TopKRouter(d_model, n_experts, k)
        self.experts = nn.ModuleList(
            [SwiGLUExpert(d_model, d_ff) for _ in range(n_experts)]
        )
        # Shared experts (DeepSeek-style): always active for every token
        self.shared = nn.ModuleList(
            [SwiGLUExpert(d_model, d_ff) for _ in range(n_shared)]
        )

    def forward(self, x: torch.Tensor):
        B, T, D = x.shape
        x_flat = x.reshape(B * T, D)                          # (BT, D)

        topk_idx, topk_w, full_probs, logits = self.router(x_flat)
        # topk_idx: (BT, k)   topk_w: (BT, k)
        # full_probs: (BT, n_experts)   logits: (BT, n_experts)

        out = torch.zeros_like(x_flat)                        # (BT, D)
        for i, expert in enumerate(self.experts):
            # Find every (token, slot) where expert i was chosen
            mask = (topk_idx == i)                            # (BT, k) bool
            tok_idx, slot_idx = mask.nonzero(as_tuple=True)   # both (M,)
            if tok_idx.numel() == 0:
                continue                                      # this expert is dead this batch
            tokens = x_flat[tok_idx]                          # (M, D)
            weights = topk_w[tok_idx, slot_idx].unsqueeze(-1) # (M, 1)
            out.index_add_(0, tok_idx, weights * expert(tokens))  # scalar*vec, scatter-add

        # Shared experts run on every token (no routing)
        for sh in self.shared:
            out = out + sh(x_flat)

        return out.reshape(B, T, D), full_probs, logits, topk_idx

Three subtleties to read carefully. The mask.nonzero call gives us, for expert $i$, every $(token, slot)$ pair where expert $i$ appeared as the token's slot-th choice. The weights tensor for that expert is then the routing weight assigned at that exact slot — a scalar per token. The index_add_ writes the weighted expert output back to the right token row, and the add matters: a token routed to two experts will receive two writes that sum into the correct $\sum_{i \in \mathcal{T}_k(x)} g_i(x)\, E_i(x)$ output. Shared experts, when present, run on every token outside the routing loop — they always fire and always contribute $E^{\text{sh}}(x)$ to the sum.

This loop is $O(N)$ in expert count, which is fine for our $N = 8$ but would be wasteful for DeepSeek's $N = 256$. Production frameworks replace it with a permuted block-sparse matmul that issues a single GEMM. The math — what gets computed — is identical.

Load Balancing: The Reason MoE Training Actually Works

Train the model we have built so far for a hundred steps and look at expert usage. The histogram does not look uniform. Two experts will be hoarding 90% of the routed tokens, and the other six will receive almost nothing. Within another few hundred steps those starved experts have effectively stopped learning — they get no gradient signal, their weights drift, the router stops choosing them, and they become permanent dead weight on the GPU.

This is expert collapse, and it is the central failure mode of MoE training. The mechanism is a positive feedback loop. The router slightly favors expert 3 by chance. Expert 3 sees more tokens, gets more gradient updates, becomes better at processing the kinds of tokens it has seen. The router learns to send those tokens to expert 3 with even higher confidence. The other experts process fewer tokens, fall behind, and the router routes around them more aggressively. Within a few thousand steps the model has effectively become a dense network running expert 3 with two extra dead modules attached.

The fix Switch Transformer introduced is an auxiliary loss that the model trains against. Let $f_i$ be the fraction of tokens that selected expert $i$ in the current batch (a hard count, divided by $BT$), and let $P_i$ be the mean of the full router probability for expert $i$ across the batch (a soft mean of the softmax outputs):

$$\mathcal{L}_{\text{aux}} = \alpha\, N \sum_{i=1}^{N} f_i\, P_i$$

The product $f_i\, P_i$ is small when both quantities are small (an expert that is both rarely chosen and rarely confident). It is large when both are large (a hoarder). The sum is minimized when both $f_i$ and $P_i$ are uniform over $i$, in which case it equals $1/N$. The factor $N$ normalizes so the loss has unit value at the uniform optimum, and $\alpha$ is a small mixing coefficient (typically $0.01$). Crucially, $P_i$ is differentiable through the softmax, so this loss has a real gradient pulling the router toward balance — even though $f_i$ itself is not differentiable.

The second loss is the router z-loss (ST-MoE, Zoph 2022). The router's pre-softmax logits can drift to extreme values, which causes both the softmax to saturate (killing gradients to the non-top-$k$ experts) and bf16 underflow / overflow:

$$\mathcal{L}_z = \frac{1}{BT}\sum_{b,t}\Bigl(\log \textstyle\sum_{j=1}^{N} e^{x_{b,t,j}}\Bigr)^{\!2}$$

This penalizes the squared log-sum-exp of the router logits per token. It costs the model nothing in expressivity — the softmax is shift-invariant, so subtracting any constant from all logits leaves routing unchanged — but it keeps the raw logits from drifting in absolute magnitude. Mixing coefficient $\beta = 0.001$ is the published default.


def load_balance_loss(full_probs: torch.Tensor, topk_idx: torch.Tensor,
                      n_experts: int) -> torch.Tensor:
    """Switch Transformer auxiliary load-balance loss.
       full_probs: (BT, n_experts)  router softmax over all experts
       topk_idx:   (BT, k)          chosen expert ids per token
       Returns scalar; multiply by alpha (e.g. 0.01) before summing."""
    BT = full_probs.size(0)
    # f_i: fraction of tokens that picked expert i (in any of their k slots)
    one_hot = F.one_hot(topk_idx, n_experts).float()  # (BT, k, n_experts)
    f = one_hot.sum(dim=(0, 1)) / BT                  # (n_experts,)
    # P_i: mean router probability for expert i
    P = full_probs.mean(dim=0)                        # (n_experts,)
    # n_experts * sum(f_i * P_i) — minimum value is 1.0 at uniform routing
    return n_experts * (f * P).sum()

def router_z_loss(logits: torch.Tensor) -> torch.Tensor:
    """ST-MoE router z-loss. Penalizes large absolute router logits.
       logits: (BT, n_experts)
       Returns scalar; multiply by beta (e.g. 0.001) before summing."""
    log_z = torch.logsumexp(logits, dim=-1)           # (BT,)
    return (log_z ** 2).mean()

Two implementation notes that are easy to get wrong. First, $f_i$ uses both top-$k$ slots: a token routed to experts 3 and 6 contributes to $f_3$ and $f_6$. The one-hot sum across the slot dimension takes care of this. Second, $P_i$ is computed on the full softmax over all $N$ experts, not the renormalized top-$k$ weights — the load-balance signal needs to know what the router would have assigned to the unchosen experts, not just how it allocated mass among the chosen ones.

The MoE Transformer Block and the Full Model

An MoE transformer block is just a standard pre-norm transformer block with the dense FFN swapped for an MoELayer. Everything else — RMSNorm, causal multi-head attention with rotary position embeddings, the residual structure — is unchanged from a LLaMA or Mistral block (covered in detail in Chapter 8). We will write a compact version of the block here for completeness, then assemble the full language model.


class RMSNorm(nn.Module):
    def __init__(self, d: int, eps: float = 1e-6):
        super().__init__()
        self.eps = eps
        self.weight = nn.Parameter(torch.ones(d))
    def forward(self, x):
        # x: (..., d)  — element-wise normalize then per-dim scale
        rms = x.pow(2).mean(-1, keepdim=True).add(self.eps).rsqrt()
        return x * rms * self.weight                     # element-wise scale

class CausalSelfAttention(nn.Module):
    """Standard causal MHA with rotary embeddings; KV-cache aware.
    Kept brief — the focus of this chapter is the MoE FFN."""
    def __init__(self, d_model, n_heads):
        super().__init__()
        self.n_heads = n_heads
        self.d_head = d_model // n_heads
        self.qkv = nn.Linear(d_model, 3 * d_model, bias=False)
        self.proj = nn.Linear(d_model, d_model, bias=False)
    def forward(self, x, kv_cache=None, rope_cos=None, rope_sin=None):
        B, T, D = x.shape
        q, k, v = self.qkv(x).chunk(3, dim=-1)
        q = q.view(B, T, self.n_heads, self.d_head).transpose(1, 2)
        k = k.view(B, T, self.n_heads, self.d_head).transpose(1, 2)
        v = v.view(B, T, self.n_heads, self.d_head).transpose(1, 2)
        q, k = apply_rope(q, k, rope_cos, rope_sin)      # rotary on q, k
        if kv_cache is not None:
            k = torch.cat([kv_cache[0], k], dim=2)       # extend along T
            v = torch.cat([kv_cache[1], v], dim=2)
            new_cache = (k, v)
        else:
            new_cache = (k, v)
        out = F.scaled_dot_product_attention(q, k, v, is_causal=(kv_cache is None))
        return self.proj(out.transpose(1, 2).reshape(B, T, D)), new_cache

class MoEBlock(nn.Module):
    def __init__(self, d_model, n_heads, d_ff, n_experts, k, n_shared, use_moe):
        super().__init__()
        self.norm1 = RMSNorm(d_model)
        self.attn  = CausalSelfAttention(d_model, n_heads)
        self.norm2 = RMSNorm(d_model)
        self.use_moe = use_moe
        if use_moe:
            self.ffn = MoELayer(d_model, d_ff, n_experts, k, n_shared)
        else:
            self.ffn = SwiGLUExpert(d_model, d_ff)       # dense fallback

    def forward(self, x, kv_cache=None, rope_cos=None, rope_sin=None):
        h, new_cache = self.attn(self.norm1(x), kv_cache, rope_cos, rope_sin)
        x = x + h
        if self.use_moe:
            ffn_out, full_probs, logits, topk_idx = self.ffn(self.norm2(x))
            x = x + ffn_out
            return x, new_cache, (full_probs, logits, topk_idx)
        else:
            x = x + self.ffn(self.norm2(x))
            return x, new_cache, None

The block returns three things: the new activations, the updated KV cache, and (when this layer is an MoE layer) a tuple of routing artifacts that the training loop needs for the auxiliary losses. Dense layers return None for that third slot. The apply_rope helper is the standard rotary embedding application from Chapter 8 — we omit its body for space but it is a few lines of broadcasting against $\cos$ / $\sin$ tables.

The full language model stitches a stack of these blocks between an embedding and an LM head. We follow the common practice of alternating dense and MoE layers — this is what Mixtral and DeepSeek both do, with MoE on the odd-indexed layers in our config. Alternation reduces total parameters slightly and gives the dense layers a chance to mix information across tokens before the next routing decision.


class MoELM(nn.Module):
    def __init__(self, vocab_size, d_model=256, n_heads=4, d_ff=1024,
                 n_layers=6, n_experts=8, k=2, n_shared=1, max_seq_len=512):
        super().__init__()
        self.tok_emb = nn.Embedding(vocab_size, d_model)
        # Alternate: even layers are dense, odd layers are MoE
        self.blocks = nn.ModuleList([
            MoEBlock(d_model, n_heads, d_ff, n_experts, k, n_shared,
                     use_moe=(layer_idx % 2 == 1))
            for layer_idx in range(n_layers)
        ])
        self.norm = RMSNorm(d_model)
        self.head = nn.Linear(d_model, vocab_size, bias=False)
        self.head.weight = self.tok_emb.weight              # weight tying
        self.rope_cos, self.rope_sin = build_rope(max_seq_len, d_model // n_heads)

    def forward(self, tokens, kv_caches=None):
        # tokens: (B, T) int64
        x = self.tok_emb(tokens)                            # (B, T, d_model)
        T = tokens.size(1)
        cos, sin = self.rope_cos[:T].to(x.device), self.rope_sin[:T].to(x.device)
        new_caches, routing = [], []
        for i, block in enumerate(self.blocks):
            cache_in = kv_caches[i] if kv_caches is not None else None
            x, new_cache, route_info = block(x, cache_in, cos, sin)
            new_caches.append(new_cache)
            if route_info is not None:
                routing.append(route_info)                  # one entry per MoE layer
        x = self.norm(x)
        logits = self.head(x)                               # (B, T, vocab_size)
        return logits, new_caches, routing

The full model with our reference config (d_model=256, n_heads=4, d_ff=1024, n_layers=6, n_experts=8, k=2, n_shared=1, GPT-2 BPE vocab of 50,257) lands at roughly 18M total parameters and 6M active per token. That is small enough to train quickly on a single GPU and large enough that expert specialization actually emerges visibly. Bigger configs are a one-line change.

Training: Data, Loop, and the Three-Loss Balancing Act

TinyStories (Eldan and Li, 2023) is the perfect dataset for this chapter. It is roughly 2 million GPT-3.5-generated short stories with a deliberately limited vocabulary — designed to be coherent at small model scales. A 10M-parameter model trained on TinyStories produces grammatical, narratively consistent text, whereas the same model on WikiText would be incoherent. The combination of small vocabulary, narrative structure, and consistent style also makes expert specialization easy to spot: experts will visibly partition along punctuation, dialogue verbs, character names, and connectives.

We load TinyStories with HuggingFace datasets (the only HuggingFace abstraction this chapter uses — everything model-side is pure PyTorch), tokenize with the GPT-2 BPE tokenizer for vocabulary parity with real-world LMs, and pack tokens into fixed-length sequences of 512. Packing means we concatenate documents end-to-end with an end-of-text token between them, then chop the resulting stream into uniform 512-token chunks. This is dramatically more compute-efficient than padding short stories.


from datasets import load_dataset
from tokenizers import Tokenizer
from torch.utils.data import IterableDataset, DataLoader

class PackedTinyStories(IterableDataset):
    """Stream TinyStories, tokenize, and yield fixed-length token chunks."""
    def __init__(self, split: str, tokenizer: Tokenizer, seq_len: int = 512):
        self.split = split
        self.tok = tokenizer
        self.seq_len = seq_len
        self.eot = tokenizer.token_to_id("<|endoftext|>")

    def __iter__(self):
        ds = load_dataset("roneneldan/TinyStories", split=self.split, streaming=True)
        buf: list[int] = []
        for ex in ds:
            ids = self.tok.encode(ex["text"]).ids + [self.eot]
            buf.extend(ids)
            while len(buf) >= self.seq_len + 1:
                # Yield (input, target) where target is shifted by 1
                chunk = buf[: self.seq_len + 1]
                yield (torch.tensor(chunk[:-1], dtype=torch.long),
                       torch.tensor(chunk[1:],  dtype=torch.long))
                buf = buf[self.seq_len:]                    # keep tail for next chunk

# Usage
tokenizer = Tokenizer.from_pretrained("gpt2")
train_ds = PackedTinyStories("train", tokenizer, seq_len=512)
loader = DataLoader(train_ds, batch_size=32, num_workers=2)

The training loop combines three losses: cross-entropy language modeling loss, the load-balance auxiliary loss summed over all MoE layers, and the router z-loss summed over all MoE layers. The combined scalar drives one backward pass.


def train_step(model, batch, optimizer, scaler, scheduler,
               alpha=0.01, beta=0.001, grad_clip=1.0):
    """One training step: forward, three-loss combine, backward, step.
    Returns dict of scalar diagnostics for logging."""
    inputs, targets = batch                                # both (B, T)
    inputs, targets = inputs.cuda(), targets.cuda()

    optimizer.zero_grad(set_to_none=True)
    with torch.autocast(device_type="cuda", dtype=torch.bfloat16):
        logits, _, routing = model(inputs)                 # (B, T, V), [], list of (probs, logits, idx)

        # 1. Language modeling loss
        lm_loss = F.cross_entropy(
            logits.reshape(-1, logits.size(-1)), targets.reshape(-1)
        )

        # 2. + 3. Sum aux + z losses across all MoE layers in this forward pass
        aux = sum(load_balance_loss(p, idx, model.blocks[0].ffn.n_experts)
                  for (p, _, idx) in routing) if routing else 0.0
        zl  = sum(router_z_loss(lg) for (_, lg, _) in routing) if routing else 0.0

        total = lm_loss + alpha * aux + beta * zl

    scaler.scale(total).backward()
    scaler.unscale_(optimizer)
    torch.nn.utils.clip_grad_norm_(model.parameters(), grad_clip)
    scaler.step(optimizer)
    scaler.update()
    scheduler.step()

    # Log expert usage histogram from the last MoE layer for monitoring
    with torch.no_grad():
        last_idx = routing[-1][2]                          # (BT, k)
        usage = torch.bincount(last_idx.flatten(),
                               minlength=model.blocks[0].ffn.n_experts).float()
        usage = usage / usage.sum()
    return {"lm": lm_loss.item(), "aux": float(aux), "z": float(zl),
            "total": total.item(), "usage": usage.cpu()}

A handful of practical knobs matter. Use bfloat16 autocast on Ampere or newer (A100, H100, RTX 3090/4090) — bf16 has the same exponent range as fp32 and avoids the GradScaler dance that fp16 needs, though we keep GradScaler in the code for fp16 fallback. Gradient clipping at norm 1.0 is helpful but not as critical as for dense models — the router z-loss already controls the worst logit drift. AdamW with $\beta_1 = 0.9, \beta_2 = 0.95, \epsilon = 10^{-8}$, weight decay 0.1 (excluding norms and the router gate from decay), peak LR $3 \times 10^{-4}$ with 200 steps of linear warmup followed by cosine to 10% of peak, total 4000 steps. Budget: roughly 50 minutes on a single A100, somewhat longer on a 3090.

Set $\alpha$ and $\beta$ at the published defaults (0.01 and 0.001) and leave them. The temptation to push $\alpha$ higher because expert usage looks unbalanced is almost always wrong: at $\alpha = 0.1$ the model spends so much capacity on satisfying the auxiliary objective that the LM loss visibly stalls. Better routing balance is the goal; sacrificing modeling quality to get there is not.

Inference, Evaluation, and Expert Forensics

Generation from an MoE model has the same KV-cache structure as generation from a dense model. The router runs per token at each step; the chosen experts execute their forward passes; the rest sit idle. The KV cache lives on the attention sublayer — it is unaffected by the MoE machinery on the FFN sublayer. We will write a generator that streams tokens with top-$p$ sampling and, as a debugging aid, logs which experts fired for each emitted token.


@torch.no_grad()
def generate(model, prompt_ids, max_new_tokens=128, top_p=0.9, temperature=1.0):
    """Autoregressive generation with KV cache and per-step routing log."""
    model.eval()
    tokens = prompt_ids.clone().cuda()                     # (1, T0)
    kv_caches = None                                        # built up step by step
    routing_log = []                                        # list of (n_moe_layers, k) per step

    # Process the prompt once to populate the cache
    logits, kv_caches, routing = model(tokens, kv_caches=None)
    routing_log.append([r[2][-1].cpu() for r in routing])  # last token's expert ids per layer

    for _ in range(max_new_tokens):
        next_logits = logits[:, -1, :] / temperature       # (1, V)
        # top-p (nucleus) sampling
        sorted_logits, sorted_idx = next_logits.sort(descending=True)
        probs = F.softmax(sorted_logits, dim=-1)
        cumprobs = probs.cumsum(dim=-1)
        cutoff = (cumprobs > top_p).float().argmax(dim=-1).clamp(min=1)
        sorted_logits[:, cutoff:] = float("-inf")
        sample = torch.multinomial(F.softmax(sorted_logits, dim=-1), 1)
        next_tok = sorted_idx.gather(-1, sample)            # (1, 1)
        tokens = torch.cat([tokens, next_tok], dim=1)

        # Forward only the new token; KV cache extends inside attention
        logits, kv_caches, routing = model(next_tok, kv_caches=kv_caches)
        routing_log.append([r[2][-1].cpu() for r in routing])

    return tokens, routing_log

The routing_log structure is what makes MoE inference forensically interesting. For each generated token we capture, for each MoE layer, the indices of the experts that fired. Aligning this against the decoded token stream gives a routing trace: "the period at position 47 was processed by experts 3 and 7 in layer 1 and experts 2 and 5 in layer 3". Aggregated across many generations, this reveals the latent specialization the model learned.

Evaluation needs three quantities. Perplexity on a held-out validation split measures modeling quality — the standard LM benchmark. Router entropy measures how spread or peaked the routing distribution is on average; lower entropy means more decisive routing. Dead-expert fraction counts how many experts were never selected by any token in the eval batch. Load imbalance coefficient is the ratio of max to mean usage; perfect balance is 1.0, total collapse approaches $N$.


@torch.no_grad()
def evaluate(model, val_loader, n_experts: int):
    """Returns perplexity + router diagnostics. Run on val split."""
    model.eval()
    total_tokens, total_nll = 0, 0.0
    usage_counts = torch.zeros(n_experts)
    entropy_sum, entropy_count = 0.0, 0

    for inputs, targets in val_loader:
        inputs, targets = inputs.cuda(), targets.cuda()
        logits, _, routing = model(inputs)
        nll = F.cross_entropy(
            logits.reshape(-1, logits.size(-1)), targets.reshape(-1),
            reduction="sum"
        )
        total_nll += nll.item()
        total_tokens += targets.numel()

        for full_probs, _, idx in routing:                  # one entry per MoE layer
            # Count how often each expert was chosen (across both top-k slots)
            usage_counts += torch.bincount(idx.flatten().cpu(),
                                           minlength=n_experts).float()
            # Mean entropy of router distribution per token: -sum(p log p)
            ent = -(full_probs * (full_probs.clamp_min(1e-12)).log()).sum(-1)
            entropy_sum += ent.sum().item()
            entropy_count += ent.numel()

    perplexity = torch.exp(torch.tensor(total_nll / total_tokens)).item()
    usage_frac = usage_counts / usage_counts.sum()
    dead_frac = (usage_counts == 0).float().mean().item()
    load_imbalance = usage_frac.max().item() / usage_frac.mean().item()
    avg_entropy = entropy_sum / entropy_count
    return {"perplexity": perplexity, "dead_expert_frac": dead_frac,
            "load_imbalance": load_imbalance, "router_entropy": avg_entropy,
            "usage_per_expert": usage_frac.tolist()}

Expected numbers after the full 4000-step training run on TinyStories: validation perplexity around 4.5 (TinyStories is intentionally easy — this number is competitive with much larger dense models on this benchmark). Dead-expert fraction at zero. Load imbalance coefficient around 1.3–1.6 (max expert sees about 50% more traffic than average, well within the “healthy” regime). Router entropy around 1.4–1.7 nats (the maximum possible at $N = 8$ is $\log 8 \approx 2.08$ nats; values around 1.5 mean confident but not collapsed routing). A dense baseline with the same active-parameter budget — same $d_{model}$, $d_{ff}$, $n_{layers}$ but with regular SwiGLU FFNs throughout — lands around perplexity 5.1, giving the MoE roughly 13% lower perplexity at equal inference cost.

The most rewarding diagnostic is the expert-utilization heatmap. Bin the validation tokens into linguistic categories — punctuation, function words, common nouns, verbs, dialogue markers, character names — and look at which experts each category routes to. The model was not told these categories existed. This is the payoff of the chapter. Specialization emerges. The router learned, with no guidance beyond next-token cross-entropy and the load-balance auxiliary, to send punctuation to one expert, dialogue verbs to another, and connectives to a third. The same phenomenon at much larger scale is what makes Mixtral and DeepSeek work — experts in production MoEs partition along syntactic, semantic, and even language boundaries (in multilingual models). The mechanism we built in this chapter is the entire mechanism, just smaller.

From here, three directions extend the project naturally. Fine-grained experts (DeepSeek's recipe) splits each expert into smaller pieces and increases $k$ proportionally — instead of 8 experts top-2, run 64 experts top-16 with the same total parameter budget. The finer granularity gives the router more freedom to compose specializations. Multi-head Latent Attention (MLA) compresses the KV cache into a low-rank latent space, dramatically cutting the memory footprint at long context. Auxiliary-loss-free routing (DeepSeek V3) replaces the auxiliary loss with a per-expert bias correction, removing the small modeling-quality cost that aux loss imposes. All three are bolt-on additions to the model we just built — the scatter-gather dispatch, the SwiGLU expert, the KV-cache structure all carry over unchanged. Implementing them is the natural next step for a reader who wants to understand modern frontier-scale sparse architectures end to end.
