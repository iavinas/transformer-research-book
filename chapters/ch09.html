Chapter 09

Encoder-Decoder Models

Chapters 7 and 8 drew a sharp line between two Transformer variants: encoder-only models that understand but cannot generate, and decoder-only models that generate but process input and output in a single undifferentiated stream. There is a third option — the original one. The Transformer was born as an encoder-decoder, with a dedicated encoder that builds a rich bidirectional representation of the input and a separate decoder that autoregressively produces the output while cross-attending to that representation. This architecture is a natural fit for any task with a clearly defined input and a structurally different output: translation, summarization, question generation, data-to-text. By the end of this chapter you will understand how T5 unified every NLP task into a single text-to-text format, how BART learned to reconstruct aggressively corrupted text, why encoder-decoder models still dominate certain conditional generation benchmarks, and when you should reach for this architecture instead of a decoder-only LLM.

The Encoder-Decoder Architecture

Before diving into specific models, it helps to recall the structural idea. An encoder-decoder Transformer has two stacks of layers. The encoder stack processes the entire input bidirectionally — every token can attend to every other token, exactly like BERT. The decoder stack generates the output autoregressively — each token can only attend to previously generated tokens, exactly like GPT. The bridge between them is cross-attention: every decoder layer contains an additional attention sublayer where the queries come from the decoder's hidden states but the keys and values come from the encoder's final output.

Concretely, if the encoder output is $H_\text{enc} \in \mathbb{R}^{n \times d}$ (where $n$ is the source length) and the decoder's hidden state at layer $l$ is $H_\text{dec}^{(l)} \in \mathbb{R}^{m \times d}$ (where $m$ is the target length generated so far), then cross-attention computes:

$$Q = H_\text{dec}^{(l)} W_Q, \quad K = H_\text{enc}\, W_K, \quad V = H_\text{enc}\, W_V$$

Here each projection is a matrix multiply (e.g., $H_\text{dec}^{(l)} W_Q$, with $W_Q \in \mathbb{R}^{d \times d_k}$, yielding $Q \in \mathbb{R}^{m \times d_k}$). The attention scores are then:

$$\text{CrossAttn}(Q, K, V) = \text{softmax}\!\left(\frac{Q K^\top}{\sqrt{d_k}}\right) V$$

The product $QK^\top$ (matrix multiply, $\mathbb{R}^{m \times d_k} \times \mathbb{R}^{d_k \times n} \to \mathbb{R}^{m \times n}$) produces an attention matrix where each of the $m$ decoder positions distributes weight over all $n$ encoder positions. This is the mechanism that lets the decoder "look back" at the full input while generating each output token.

The critical advantage of this design is that the encoder processes the input once, and the decoder can attend to that full representation at every generation step. In a decoder-only model, the input and output share the same attention budget: the model must fit "understand the input" and "generate the output" into a single left-to-right pass. The encoder-decoder architecture decouples these two responsibilities.

Why Cross-Attention Matters

Cross-attention lets the decoder dynamically select which parts of the input are relevant to the current generation step. When translating "Le chat est assis sur le tapis" to English and generating the word "sitting," the cross-attention layer learns to place high weight on "assis" (the French word for sitting) and lower weight on the article "le." This soft alignment emerges from training — no explicit word alignment is provided.

Compare this to a decoder-only model handling the same task. The decoder-only approach concatenates the source and target into a single sequence: "Le chat est assis sur le tapis [SEP] The cat is sitting on the mat." Every source token occupies positions in the same causal attention window as the target tokens. This works, but it means the source tokens can only attend to other source tokens to their left — not bidirectionally. The encoder-decoder design gives the source its own bidirectional processing, which is strictly more powerful for understanding the input.

T5 (Raffel et al., 2020) — Text-to-Text Transfer Transformer

T5 is not just a model — it is a systematic study of what works in transfer learning. The paper "Exploring the Limits of Transfer Learning with a Unified Text-to-Text Transformer" ran hundreds of experiments comparing pre-training objectives, architectures, dataset sizes, and fine-tuning strategies. The conclusion was a single model trained under a single paradigm: every task — classification, regression, translation, summarization, question answering — is formatted as a text-to-text problem. The input is a string. The output is a string. Nothing else.

The Text-to-Text Unification

The idea is to prepend a task-specific prefix to the input and train the model to produce the target text. No task-specific heads. No classification layers. No span extractors. Just string in, string out.

This unification eliminates an entire category of engineering decisions. Instead of designing a classification head for sentiment, a span extraction module for QA, and a sequence generation decoder for summarization, you just train one model on all tasks simultaneously. The task prefix acts as a soft routing mechanism — the model learns internally how to handle each task type.

The C4 Dataset

T5 was pre-trained on the Colossal Clean Crawled Corpus (C4), a 750GB cleaned version of Common Crawl. The cleaning pipeline removed duplicate pages, pages with bad words, pages containing placeholder text ("lorem ipsum"), pages with too little text, and pages not classified as English by a language detector. C4 became a widely used pre-training corpus and demonstrated that data quality matters as much as quantity — a lesson later reinforced by the RefinedWeb and Dolma datasets.

Span Corruption Objective

T5's pre-training objective is span corruption (also called span denoising). Instead of masking individual tokens like BERT, T5 selects multiple random spans of tokens, replaces each span with a single sentinel token (e.g., <extra_id_0>, <extra_id_1>), and trains the model to output the missing spans delimited by the same sentinels.

The default configuration corrupts 15% of input tokens with an average span length of 3. This means roughly 5% of the tokens become sentinels in the input, and the target sequence is much shorter than the input — just the missing spans. Compared to BERT's single-token masking, span corruption forces the model to learn multi-token dependencies and produces shorter targets, which speeds up training.

Architecture Choices

T5 uses a standard encoder-decoder Transformer with a few notable design decisions:

  • Relative positional bias. Instead of absolute positional embeddings, T5 uses a learned relative position bias that is added to the attention logits. Each attention head learns a set of scalar biases indexed by the relative distance between query and key positions. This means the model generalizes more naturally to sequence lengths not seen during training.
  • Pre-norm. Layer normalization is applied before each sublayer (attention and FFN), not after. This is the same pre-norm variant used by GPT-2 and later models, which improves training stability.
  • No bias terms in dense layers. The linear projections in T5 omit bias vectors, following the simplification trend that later became standard in LLaMA and other modern architectures.
  • Shared embeddings. The input embedding matrix, the decoder output embedding matrix, and the final output projection matrix are all tied (shared), reducing parameter count significantly.

Here is a minimal T5-style encoder-decoder in pure PyTorch, showing the core structure:


import torch
import torch.nn as nn
import torch.nn.functional as F

class T5RelativeAttention(nn.Module):
    """Simplified T5-style multi-head attention with relative position bias."""
    def __init__(self, d_model, n_heads, max_rel_dist=128, is_cross=False):
        super().__init__()
        self.n_heads = n_heads
        self.d_k = d_model // n_heads
        self.is_cross = is_cross
        self.W_q = nn.Linear(d_model, d_model, bias=False)
        self.W_k = nn.Linear(d_model, d_model, bias=False)
        self.W_v = nn.Linear(d_model, d_model, bias=False)
        self.W_o = nn.Linear(d_model, d_model, bias=False)
        if not is_cross:  # relative bias only for self-attention
            self.rel_bias = nn.Embedding(2 * max_rel_dist + 1, n_heads)
            self.max_rel_dist = max_rel_dist

    def forward(self, x, context=None, mask=None):
        B, S, _ = x.shape
        q = self.W_q(x).view(B, S, self.n_heads, self.d_k).transpose(1, 2)  # (B, H, S, d_k)
        kv_input = context if self.is_cross and context is not None else x
        T = kv_input.shape[1]
        k = self.W_k(kv_input).view(B, T, self.n_heads, self.d_k).transpose(1, 2)  # (B, H, T, d_k)
        v = self.W_v(kv_input).view(B, T, self.n_heads, self.d_k).transpose(1, 2)  # (B, H, T, d_k)

        scores = q @ k.transpose(-2, -1) / (self.d_k ** 0.5)  # (B, H, S, T)
        if not self.is_cross:
            pos_q = torch.arange(S, device=x.device).unsqueeze(1)    # (S, 1)
            pos_k = torch.arange(T, device=x.device).unsqueeze(0)    # (1, T)
            rel = (pos_q - pos_k).clamp(-self.max_rel_dist, self.max_rel_dist)
            rel = rel + self.max_rel_dist                             # shift to [0, 2*max]
            bias = self.rel_bias(rel).permute(2, 0, 1).unsqueeze(0)  # (1, H, S, T)
            scores = scores + bias
        if mask is not None:
            scores = scores.masked_fill(mask == 0, float('-inf'))
        attn = F.softmax(scores, dim=-1)
        out = (attn @ v).transpose(1, 2).reshape(B, S, -1)           # (B, S, d_model)
        return self.W_o(out)

class T5Block(nn.Module):
    """Single T5 encoder or decoder block (pre-norm)."""
    def __init__(self, d_model, n_heads, d_ff, is_decoder=False):
        super().__init__()
        self.self_attn = T5RelativeAttention(d_model, n_heads)
        self.norm1 = nn.RMSNorm(d_model)
        self.ffn = nn.Sequential(
            nn.Linear(d_model, d_ff, bias=False),
            nn.GELU(),
            nn.Linear(d_ff, d_model, bias=False),
        )
        self.norm2 = nn.RMSNorm(d_model)
        self.is_decoder = is_decoder
        if is_decoder:
            self.cross_attn = T5RelativeAttention(d_model, n_heads, is_cross=True)
            self.norm_cross = nn.RMSNorm(d_model)

    def forward(self, x, enc_out=None, self_mask=None, cross_mask=None):
        x = x + self.self_attn(self.norm1(x), mask=self_mask)
        if self.is_decoder and enc_out is not None:
            x = x + self.cross_attn(self.norm_cross(x), context=enc_out, mask=cross_mask)
        x = x + self.ffn(self.norm2(x))
        return x

Flan-T5: Instruction Fine-Tuning

Flan-T5 (Chung et al., 2022) takes a pre-trained T5 checkpoint and fine-tunes it on a mixture of over 1,800 tasks phrased as natural language instructions. The result is dramatically better zero-shot and few-shot performance — Flan-T5-XL (3B) matches or beats the original T5-11B on many benchmarks. Instruction tuning teaches the model to follow task descriptions it has never seen during fine-tuning, demonstrating that format diversity during fine-tuning is more important than scale alone.

Flan-T5 showed the encoder-decoder format is not obsolete in the instruction-following era. For tasks with clear input-output structure (summarization, translation, extraction), Flan-T5 often outperforms instruction-tuned decoder-only models of the same parameter count because the bidirectional encoder gives it a deeper understanding of the input.

BART (Lewis et al., 2020)

If T5 is a systematic study of transfer learning, BART is a specific bet: train an encoder-decoder as a denoising autoencoder with aggressive corruption, and you will get a model that excels at generation tasks, especially summarization. The intuition is simple — if a model can learn to reconstruct text that has been randomly masked, deleted, shuffled, and rotated, it must develop a deep understanding of language structure.

The Denoising Autoencoder Framework

BART takes an input document, corrupts it using one of several noise functions, feeds the corrupted version into the encoder, and trains the decoder to output the original uncorrupted text. The decoder sees the full original text as the target — unlike T5 which only outputs the missing spans. This means BART's decoder learns to generate complete, coherent text, which is directly useful for generation tasks.

Corruption Strategies

BART experiments with five corruption schemes and finds that combining them works best:

The winning combination is text infilling with span lengths drawn from a Poisson distribution ($\lambda = 3$), combined with sentence permutation. Text infilling is strictly harder than token masking because a single [MASK] token can replace 0, 1, or many tokens — the model must learn to predict how many tokens are missing, not just which ones.

Architecture

BART uses a standard Transformer encoder-decoder. BART-Base has 6 encoder layers and 6 decoder layers with $d_\text{model} = 768$, matching BERT-Base's size. BART-Large uses 12 layers each with $d_\text{model} = 1024$, matching BERT-Large. Unlike T5, BART uses learned absolute positional embeddings and includes bias terms in its linear layers.

The key difference from T5 is in the decoder target: T5's decoder only outputs the corrupted spans (short targets), while BART's decoder outputs the entire original document (long targets). This makes BART's pre-training more expensive per example but teaches the decoder to produce fluent, complete text.

BART for Summarization

BART set new state-of-the-art results on CNN/DailyMail and XSum summarization benchmarks. The pre-training is a natural match: the model already learned to produce coherent text from a degraded version of it. Summarization is just another form of reconstruction — produce clean, condensed text from a long, noisy input.

For fine-tuning on summarization, the encoder receives the full document, and the decoder generates the summary. No architectural changes are needed — the same encoder-decoder structure used during pre-training is used for the downstream task.

mBART: Multilingual BART

mBART (Liu et al., 2020) extends BART to 25 languages by pre-training on a multilingual corpus with the same denoising objective. The innovation is that mBART pre-trains on monolingual data in each language separately — no parallel sentences are needed. When fine-tuned on translation with just a few thousand parallel sentences, mBART achieves strong results, especially for low-resource language pairs. The model learns cross-lingual structure during pre-training purely from the shared denoising objective, then leverages it during fine-tuning.

Other Encoder-Decoder Models

T5 and BART defined the two poles of encoder-decoder pre-training: systematic ablation (T5) versus aggressive denoising (BART). Several subsequent models explored the space between them, each introducing a distinctive idea.

UL2: Mixture of Denoisers

UL2 (Tay et al., 2022) argues that no single pre-training objective is optimal for all downstream tasks. Short-span corruption (like T5's) helps with understanding tasks. Long-span corruption helps with generation. Causal language modeling (like GPT's) helps with open-ended generation. So why not use all three?

UL2 introduces Mixture of Denoisers (MoD), which alternates between three denoising modes during pre-training:

Each training example is tagged with a mode sentinel token ([R], [S], or [X]) so the model knows which denoiser produced the corruption. At inference time, the appropriate mode token is prepended to activate the right "skill." UL2 demonstrated that mixing objectives produces a model that is simultaneously good at understanding and generation — closing the gap between encoder-decoder and decoder-only architectures.

mT5: Multilingual T5

mT5 (Xue et al., 2021) applies the T5 recipe to 101 languages using the mC4 dataset — a multilingual version of C4 covering web text in 101 languages. The architecture and pre-training objective are identical to T5. The key contribution is demonstrating that the text-to-text framework works across languages without any architectural changes. mT5 achieves strong results on cross-lingual benchmarks like XTREME, though it requires more parameters to match the English-only performance of T5 because the model's capacity must be spread across many languages.

PEGASUS: Pre-Training for Abstractive Summarization

PEGASUS (Zhang et al., 2020) takes a targeted approach: design a pre-training objective specifically optimized for summarization. The key idea is Gap Sentence Generation (GSG) — instead of masking random tokens or spans, mask entire sentences. The selected sentences are those most similar to the rest of the document (measured by ROUGE), on the theory that these "principal" sentences approximate what a summary would contain.

The encoder receives the document with principal sentences replaced by [MASK] tokens. The decoder generates the masked sentences, concatenated together. This pre-training directly mirrors summarization: understand the document, then generate the key sentences. PEGASUS achieved state-of-the-art results on 12 summarization datasets, often with very little fine-tuning data — as few as 1,000 examples were sufficient for competitive performance.

ProphetNet: Future N-Gram Prediction

ProphetNet (Qi et al., 2020) modifies the decoder's training objective. Instead of predicting only the next token $y_t$ at each position, ProphetNet predicts the next $n$ tokens $(y_t, y_{t+1}, \ldots, y_{t+n-1})$ simultaneously using $n$ separate prediction heads that share the same decoder hidden state. With $n = 2$, each decoder position must produce a representation that is informative enough to predict both the immediate next token and the one after it.

This future n-gram prediction objective acts as a regularizer that forces the decoder to plan ahead rather than greedily optimizing for the next token. The result is improved coherence in long-form generation, with particular gains on summarization and question generation tasks where future planning matters most.

Encoder-Decoder vs. Encoder-Only vs. Decoder-Only

With three chapters of context — BERT (encoder-only), GPT (decoder-only), and T5/BART (encoder-decoder) — we can now compare these architectures systematically. The choice is not about which is "best" in absolute terms but about which matches the structure of your task.

When to Use Encoder-Decoder

The encoder-decoder architecture is the right choice when your task has all three of these properties:

  1. Clearly defined input. There is a concrete source document — a passage to summarize, a sentence to translate, a table to verbalize, a question to answer from a context. The input is not open-ended.
  2. Structurally different output. The output is a different kind of text than the input — a summary, a translation, a generated question, a data description. The output is not just a continuation of the input.
  3. Input understanding matters more than generation flexibility. The task rewards deep comprehension of the input more than creative or open-ended generation. Summarization quality depends on understanding the source; chatbot quality depends on conversational fluency.

For translation, summarization, question generation, data-to-text, and speech-to-text, encoder-decoder models consistently match or outperform decoder-only models of the same parameter count. The bidirectional encoder provides a genuine advantage when the task requires faithful comprehension of a fixed input.

For general chat, open-ended generation, in-context learning, and tasks where the boundary between "input" and "output" is blurry, decoder-only models are more natural. The decoder-only architecture also benefits from simpler scaling — you only scale one stack of layers, not two — and from a larger ecosystem of infrastructure (KV caching, speculative decoding, quantization) optimized for autoregressive models.

Key Insight

The encoder-decoder architecture separates input understanding from output generation, letting each use the attention pattern best suited to its role. The encoder processes the input bidirectionally (like BERT), while the decoder generates the output autoregressively (like GPT). This structural decomposition is why encoder-decoder models punch above their weight on conditional generation tasks — they allocate dedicated capacity to understanding the input, rather than splitting attention between comprehension and generation in a single pass.

Training an Encoder-Decoder from Scratch

To make the encoder-decoder architecture concrete, let us build a minimal training loop that pre-trains a small model on the span corruption objective and then fine-tunes it on summarization.

First, the span corruption data preparation — we implement T5-style span masking:


import torch
import torch.nn as nn
import random

def create_span_corruption(token_ids, vocab_size, mask_ratio=0.15, mean_span_len=3):
    """T5-style span corruption: replace random spans with sentinel tokens.

    Args:
        token_ids: 1D tensor of token IDs
        vocab_size: size of vocabulary (sentinels start at vocab_size - 100)
        mask_ratio: fraction of tokens to mask
        mean_span_len: average span length (Poisson lambda)
    Returns:
        input_ids: corrupted input with sentinels
        target_ids: sentinel-delimited missing spans
    """
    n = len(token_ids)
    n_mask = max(1, int(n * mask_ratio))

    # Sample span lengths from geometric distribution (approximates Poisson)
    spans, total = [], 0
    while total < n_mask:
        span_len = max(1, int(random.expovariate(1.0 / mean_span_len)))
        spans.append(span_len)
        total += span_len

    # Sample non-overlapping span start positions
    mask = torch.zeros(n, dtype=torch.bool)
    starts = sorted(random.sample(range(max(1, n - max(spans))), min(len(spans), n // mean_span_len)))

    sentinel_id = vocab_size - 1  # first sentinel
    input_tokens, target_tokens = [], []
    prev_end, sentinel_idx = 0, 0

    for i, start in enumerate(starts):
        if start < prev_end:
            continue
        end = min(start + spans[i % len(spans)], n)
        # Add uncorrupted tokens before this span
        input_tokens.extend(token_ids[prev_end:start].tolist())
        # Add sentinel to input
        input_tokens.append(sentinel_id - sentinel_idx)
        # Add sentinel + masked tokens to target
        target_tokens.append(sentinel_id - sentinel_idx)
        target_tokens.extend(token_ids[start:end].tolist())
        sentinel_idx += 1
        prev_end = end

    # Add remaining uncorrupted tokens
    input_tokens.extend(token_ids[prev_end:].tolist())

    return torch.tensor(input_tokens), torch.tensor(target_tokens)

Now the training loop for both pre-training and fine-tuning. Pre-training uses span corruption; fine-tuning uses standard sequence-to-sequence cross-entropy:


def train_enc_dec(model, dataloader, optimizer, epochs=3, device='cuda'):
    """Standard encoder-decoder training loop.

    Each batch contains:
        src_ids: (batch, src_len) — encoder input
        tgt_ids: (batch, tgt_len) — decoder input (teacher-forced, shifted right)
        labels:  (batch, tgt_len) — target token IDs for cross-entropy
    """
    model.train()
    for epoch in range(epochs):
        total_loss = 0
        for batch in dataloader:
            src = batch['src_ids'].to(device)           # (B, S)
            tgt = batch['tgt_ids'].to(device)           # (B, T)
            labels = batch['labels'].to(device)         # (B, T)

            # Encoder processes input bidirectionally
            enc_out = model.encode(src)                  # (B, S, d_model)
            # Decoder generates with causal mask + cross-attention to encoder
            logits = model.decode(tgt, enc_out)          # (B, T, vocab_size)

            loss = nn.functional.cross_entropy(
                logits.view(-1, logits.size(-1)),        # (B*T, vocab_size)
                labels.view(-1),                         # (B*T,)
                ignore_index=-100
            )
            optimizer.zero_grad()
            loss.backward()
            torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
            optimizer.step()
            total_loss += loss.item()

        print(f"Epoch {epoch+1} — avg loss: {total_loss / len(dataloader):.4f}")

The training loop is identical for pre-training and fine-tuning — only the data changes. During pre-training, src_ids contains the span-corrupted input and labels contains the sentinel-delimited targets. During fine-tuning on summarization, src_ids is the document and labels is the summary.

Greedy Decoding: The Encoder Runs Once

The most important operational difference between encoder-decoder and decoder-only inference is this: the encoder processes the input exactly once, and its output is reused at every decoder step. The decoder generates one token at a time, feeding each predicted token back as input for the next step, while cross-attending to the cached encoder representation.


@torch.no_grad()
def greedy_decode(model, src_ids, max_len=128, bos_id=0, eos_id=1, device='cuda'):
    """Encoder-decoder greedy decoding.

    Key insight: model.encode() runs ONCE. The decoder loops token-by-token,
    cross-attending to the same encoder output at every step.
    """
    src = src_ids.unsqueeze(0).to(device)              # (1, S)

    # --- Encoder runs once ---
    enc_out = model.encode(src)                         # (1, S, d_model)

    # --- Decoder generates autoregressively ---
    generated = [bos_id]
    for step in range(max_len):
        tgt = torch.tensor([generated], device=device)  # (1, step+1)

        # Causal self-attention over generated tokens so far,
        # then cross-attention to enc_out (same cached representation every step)
        logits = model.decode(tgt, enc_out)              # (1, step+1, vocab_size)

        next_token = logits[0, -1].argmax().item()       # greedy: pick highest prob
        generated.append(next_token)

        if next_token == eos_id:
            break

    return generated  # list of token IDs including BOS and EOS

Compare this to decoder-only generation, where the entire input prefix must be included in the decoder's context at every step. Here, the encoder output enc_out is computed once and the decoder only grows by one token per step. For long inputs (e.g., a 10,000-token document being summarized), this is substantially more efficient — the $O(n^2)$ cost of encoding the input is paid once rather than at every generation step.

The Encoder-Decoder in 2025

The industry trend has moved decisively toward decoder-only models for general-purpose systems. GPT-4, Claude, Gemini, and LLaMA are all decoder-only. The encoder-decoder architecture has not disappeared — it has specialized. It remains the standard for machine translation (NLLB, SeamlessM4T), speech recognition (Whisper uses an encoder-decoder), document understanding tasks, and any setting where the input is a fixed artifact that needs deep comprehension before generation begins.

The encoder-decoder's fundamental advantage — separate, bidirectional input processing — becomes most valuable when the input is long, structured, or in a different modality than the output. For a 10,000-word document being summarized into 200 words, the encoder processes those 10,000 words bidirectionally once, and the decoder generates the summary while cross-attending to the full representation. A decoder-only model would process all 10,200 tokens in a single causal pass, with each input token only able to attend to the tokens before it.

Understanding all three architectures — encoder-only, decoder-only, and encoder-decoder — gives you the vocabulary to make informed design decisions rather than defaulting to the most popular option.
